# 5.1 Monitor Resources by Using Azure Monitor — AZ-104 Study Notes

---

## Overview

Azure Monitor is the **comprehensive monitoring solution** in Azure. It collects, analyzes, and acts on telemetry data from Azure, on-premises, and multi-cloud environments. It's the foundation for all monitoring in Azure.

---

## 5.1.1 Azure Monitor Overview

### What is Azure Monitor?
- A **full-stack monitoring service** that collects metrics and logs from all Azure resources
- Provides visualization, analysis, alerting, and automated actions
- Integrates with: Application Insights, Log Analytics, Azure Monitor Alerts, Workbooks

### Azure Monitor Architecture

```
Data Sources → Azure Monitor → Consume
                  ├── Metrics (numeric, time-series)
                  └── Logs (structured, Log Analytics)

Data Sources:
  ├── Application (App Insights)
  ├── Guest OS (VM extensions/agents)
  ├── Azure Resources (platform metrics/logs)
  ├── Azure Subscription (activity log)
  └── Azure Tenant (Entra ID logs)
```

### Key Components

| Component | Description |
|-----------|-------------|
| **Metrics** | Numeric time-series data; collected automatically; near real-time |
| **Logs** | Structured data stored in Log Analytics; queried with KQL |
| **Alerts** | Notifications/actions when conditions are met |
| **Workbooks** | Interactive reports and dashboards |
| **Insights** | Pre-built monitoring experiences (VM Insights, Container Insights) |
| **Application Insights** | APM for web applications |

## 5.1.2 Azure Monitor Metrics

### What Are Metrics?
- **Numerical values** collected at regular intervals
- Lightweight, near real-time (1-minute granularity)
- Stored for **93 days** by default
- Automatically collected for most Azure resources

### Types of Metrics

| Type | Description | Examples |
|------|-------------|---------|
| **Platform metrics** | Auto-collected from Azure resources | CPU %, Network In/Out, Disk IOPS |
| **Guest OS metrics** | From inside the VM (requires agent) | Memory %, Disk space, Process count |
| **Application metrics** | From Application Insights | Request rate, Response time, Failure rate |
| **Custom metrics** | Sent via API or agents | Business-specific metrics |

### Metrics Explorer
- **Visualize metrics** with charts in Azure portal
- Portal: Monitor → Metrics
- Select: Resource → Metric Namespace → Metric → Aggregation

### Aggregation Types

| Type | Description |
|------|-------------|
| **Average** | Mean value over time period |
| **Sum** | Total of all values |
| **Min** | Lowest value |
| **Max** | Highest value |
| **Count** | Number of data points |

### Common VM Metrics

| Metric | Description | Type |
|--------|-------------|------|
| **Percentage CPU** | CPU utilization | Platform |
| **Network In/Out Total** | Bytes received/sent | Platform |
| **Disk Read/Write Bytes** | Disk I/O | Platform |
| **Available Memory Bytes** | Free RAM | Guest OS (needs agent) |
| **Disk Free Space** | Available disk space | Guest OS (needs agent) |

> [!IMPORTANT]
> **Memory metrics** are NOT available as platform metrics. You need the **Azure Monitor Agent** to collect guest OS metrics like available memory.

### Diagnostic Settings
- Configure where to send platform logs and metrics:
  - **Log Analytics workspace** — for querying with KQL
  - **Storage account** — for archival
  - **Event Hub** — for streaming to external systems

```bash
# Create diagnostic setting
az monitor diagnostic-settings create \
  --resource "/subscriptions/{sub}/resourceGroups/myRG/providers/Microsoft.Compute/virtualMachines/myVM" \
  --name "myDiagSetting" \
  --logs '[{"category": "Administrative", "enabled": true}]' \
  --metrics '[{"category": "AllMetrics", "enabled": true}]' \
  --workspace myLogAnalyticsWorkspace
```

## 5.1.3 Azure Monitor Logs and Log Analytics

### What is Log Analytics?
- A tool in Azure Monitor to **query and analyze log data**
- Uses **Kusto Query Language (KQL)**
- Data stored in a **Log Analytics workspace**

### Log Analytics Workspace
- **Central repository** for log data from multiple sources
- Data retention: **30 days** (default) to **730 days** (2 years)
- Multiple resources can send data to the same workspace

```bash
# Create Log Analytics workspace
az monitor log-analytics workspace create --resource-group myRG \
  --workspace-name myWorkspace --location eastus \
  --retention-time 90
```

### Common KQL Queries

```kql
// CPU usage over last 24 hours
Perf
| where ObjectName == "Processor" and CounterName == "% Processor Time"
| where TimeGenerated > ago(24h)
| summarize AvgCPU = avg(CounterValue) by bin(TimeGenerated, 1h), Computer
| render timechart

// Failed VM heartbeats (unavailable VMs)
Heartbeat
| summarize LastHeartbeat = max(TimeGenerated) by Computer
| where LastHeartbeat < ago(5m)

// Top 10 events by type
Event
| summarize Count = count() by EventLevelName
| top 10 by Count

// Sign-in failures
SigninLogs
| where ResultType != 0
| summarize FailureCount = count() by UserDisplayName, ResultDescription
| order by FailureCount desc

// Azure Activity Log - resource deletions
AzureActivity
| where OperationNameValue endswith "DELETE"
| where ActivityStatusValue == "Success"
| project TimeGenerated, Caller, ResourceGroup, Resource = _ResourceId
| order by TimeGenerated desc
```

### KQL Basics

| Operator | Description | Example |
|----------|-------------|---------|
| `where` | Filter rows | `where CounterValue > 90` |
| `summarize` | Aggregate | `summarize avg(CounterValue) by Computer` |
| `project` | Select columns | `project TimeGenerated, Computer, CounterValue` |
| `order by` | Sort | `order by CounterValue desc` |
| `top` | Limit results | `top 10 by Count` |
| `extend` | Add computed column | `extend GB = CounterValue / 1024` |
| `render` | Visualize | `render timechart` |
| `ago()` | Time relative to now | `where TimeGenerated > ago(1h)` |
| `bin()` | Time bucketing | `bin(TimeGenerated, 5m)` |
| `count()` | Count rows | `summarize count() by Category` |
| `join` | Join tables | `Table1 | join Table2 on Computer` |

> [!TIP]
> Learn basic **KQL syntax** for the exam. You won't need to write complex queries, but you should understand `where`, `summarize`, `project`, `order by`, `ago()`, and `render`.

## 5.1.4 Azure Monitor Alerts

### Alert Components

| Component | Description |
|-----------|-------------|
| **Alert Rule** | Defines the condition that triggers the alert |
| **Action Group** | Defines what happens when the alert fires |
| **Alert Instance** | A specific occurrence of a triggered alert |

### Alert Rule Types

| Type | Based On | Example |
|------|----------|---------|
| **Metric alert** | Metric threshold | CPU > 80% for 5 minutes |
| **Log search alert** | KQL query result | More than 5 errors in 15 minutes |
| **Activity log alert** | Activity log events | VM deleted, role assignment changed |
| **Smart detection** | Application Insights anomalies | Spike in failure rate |

### Alert Severity Levels

| Severity | Level | Description |
|----------|-------|-------------|
| **Sev 0** | Critical | Immediate attention required |
| **Sev 1** | Error | Needs investigation |
| **Sev 2** | Warning | Potential issue |
| **Sev 3** | Informational | For awareness |
| **Sev 4** | Verbose | Detailed info |

### Alert States

| State | Description |
|-------|-------------|
| **New** | Alert just fired; not acknowledged |
| **Acknowledged** | Someone is investigating |
| **Closed** | Issue resolved |

### Creating Alerts
```bash
# Create metric alert (CPU > 80%)
az monitor metrics alert create --resource-group myRG \
  --name "HighCPUAlert" --scopes <vm-resource-id> \
  --condition "avg Percentage CPU > 80" \
  --window-size 5m --evaluation-frequency 1m \
  --severity 2 --action myActionGroup \
  --description "CPU usage is high"

# Create activity log alert (VM deleted)
az monitor activity-log alert create --resource-group myRG \
  --name "VMDeletedAlert" \
  --condition category=Administrative and operationName=Microsoft.Compute/virtualMachines/delete \
  --action-group myActionGroup
```

### Action Groups

| Action Type | Description |
|-------------|-------------|
| **Email/SMS/Push/Voice** | Send notifications to people |
| **Azure Function** | Trigger a function |
| **Logic App** | Start a Logic App workflow |
| **Webhook** | Call an HTTP endpoint |
| **ITSM** | Create ITSM ticket |
| **Automation Runbook** | Run an Azure Automation runbook |
| **Event Hub** | Stream to Event Hub |

```bash
# Create action group
az monitor action-group create --resource-group myRG \
  --name myActionGroup --short-name myAG \
  --action email admin admin@contoso.com \
  --action sms admin1 1 5551234567
```

> [!NOTE]
> **Action groups** are reusable — one action group can be used by multiple alert rules. Create groups for different teams (ops, security, management) and share them.

## 5.1.5 Azure Monitor Agent (AMA)

### What is AMA?
- The **recommended agent** for collecting guest OS data from VMs
- Replaces legacy agents: Log Analytics Agent (MMA/OMS), Diagnostics extension, Telegraf

### Agent Comparison

| Feature | Azure Monitor Agent (AMA) | Log Analytics Agent (Legacy) |
|---------|--------------------------|----------------------------|
| **Status** | ✅ Current | ⚠️ Deprecated (Aug 2024) |
| **Data Collection** | Data Collection Rules (DCR) | Workspace configuration |
| **Multi-homing** | ✅ Multiple workspaces | Limited |
| **Filtering** | ✅ At source | ❌ All data sent |
| **Managed Identity** | ✅ Required | ❌ Key-based |

### Data Collection Rules (DCR)
- Define **what data** to collect and **where to send** it
- Can target multiple VMs
- Fine-grained filtering — collect only what you need

```bash
# Install AMA on VM
az vm extension set --resource-group myRG --vm-name myVM \
  --name AzureMonitorWindowsAgent --publisher Microsoft.Azure.Monitor \
  --enable-auto-upgrade true
```

## 5.1.6 VM Insights

### What is VM Insights?
- A **pre-built monitoring solution** for Azure VMs
- Provides: performance charts, dependency maps, health monitoring
- Requires: Azure Monitor Agent + Data Collection Rule

### Key Features

| Feature | Description |
|---------|-------------|
| **Performance** | CPU, memory, disk, network charts |
| **Map** | Visual dependency map showing connections between VMs and external services |
| **Health** | Guest health monitoring with health states |
| **At-scale** | Monitor all VMs across subscriptions |

### Enabling VM Insights
```bash
# Enable VM Insights (installs AMA + Dependency Agent)
az vm extension set --resource-group myRG --vm-name myVM \
  --name AzureMonitorWindowsAgent --publisher Microsoft.Azure.Monitor

az vm extension set --resource-group myRG --vm-name myVM \
  --name DependencyAgentWindows \
  --publisher Microsoft.Azure.Monitoring.DependencyAgent
```

## 5.1.7 Azure Activity Log

### What is the Activity Log?
- A **subscription-level log** that records all management plane operations
- Records: who did what, when, and at what scope
- Retained for **90 days** by default; send to Log Analytics for longer retention

### Activity Log Categories

| Category | Examples |
|----------|---------|
| **Administrative** | Create/update/delete operations via ARM |
| **Service Health** | Azure service incidents, maintenance |
| **Resource Health** | Health status changes for resources |
| **Alert** | Azure alert activations |
| **Autoscale** | Scale operations |
| **Recommendation** | Azure Advisor recommendations |
| **Security** | Microsoft Defender for Cloud alerts |
| **Policy** | Azure Policy operations |

### Viewing Activity Log
```bash
# View activity log
az monitor activity-log list --resource-group myRG \
  --start-time 2024-01-01 --end-time 2024-01-31 \
  --output table

# Filter by operation
az monitor activity-log list --resource-group myRG \
  --status Succeeded --caller admin@contoso.com
```

### Archiving Activity Log
- Send to **Log Analytics workspace** for querying with KQL (recommended)
- Send to **Storage account** for long-term archival
- Send to **Event Hub** for streaming to SIEM systems

```bash
# Create diagnostic setting to archive activity log
az monitor diagnostic-settings create \
  --resource "/subscriptions/{sub-id}" \
  --name "ActivityLogToLA" \
  --logs '[{"category": "Administrative", "enabled": true}, {"category": "Security", "enabled": true}]' \
  --workspace myWorkspace
```

> [!WARNING]
> Activity Log data is retained for only **90 days**. To keep data longer, create a **diagnostic setting** to send it to a Log Analytics workspace or storage account.

## 5.1.8 Azure Workbooks

### What Are Workbooks?
- **Interactive, customizable reports** in Azure Monitor
- Combine text, metrics, logs, and parameters into rich visual reports
- Templates available for common scenarios (VM performance, security, etc.)

### Key Features
- Combine metrics + logs in one report
- Interactive parameters (drop-downs, time ranges)
- Shareable across teams
- Gallery of pre-built templates

## Exam Tips — Section 5.1

- **Azure Monitor** = central monitoring platform; collects Metrics + Logs
- **Metrics** = numeric, near real-time, 93 days retention; auto-collected
- **Logs** = structured, Log Analytics workspace, KQL queries; configurable retention (30–730 days)
- **Memory metrics** are NOT platform metrics — need Azure Monitor Agent
- **Azure Monitor Agent (AMA)** replaces legacy Log Analytics Agent
- **Data Collection Rules (DCR)** define what to collect and where to send
- Know basic **KQL**: `where`, `summarize`, `project`, `ago()`, `render`
- **Alert types**: Metric, Log search, Activity log, Smart detection
- **Action groups**: Email, SMS, Function, Logic App, Webhook, Runbook — reusable
- **Activity Log** = subscription-level, 90-day retention, send to LA for longer
- **VM Insights** requires AMA + Dependency Agent for dependency maps
- **Diagnostic settings** send platform metrics/logs to: LA workspace, Storage, Event Hub
- **Workbooks** = interactive reports combining metrics, logs, and text
- Alert severity: 0 (Critical) → 4 (Verbose)
- Alert states: New → Acknowledged → Closed
