# 5.2 Implement Backup and Recovery — AZ-104 Study Notes

---

## Overview

Azure Backup and Azure Site Recovery provide comprehensive **data protection** and **disaster recovery** for Azure resources. This section covers Azure Backup for VMs, files, databases, and Azure Site Recovery for DR scenarios.

---

## 5.2.1 Azure Backup Overview

### What is Azure Backup?
- A **cloud-based backup service** that protects data from accidental deletion, corruption, and ransomware
- **Zero-infrastructure** — no on-premises backup infrastructure needed
- Supports VMs, SQL databases, Azure Files, on-premises servers, and more

### Key Benefits

| Benefit | Description |
|---------|-------------|
| **Offloading** | No on-prem backup infrastructure |
| **Scaling** | Unlimited storage, no maintenance |
| **Security** | Encryption, soft delete, MFA for critical operations |
| **App-consistent** | Application-aware snapshots |
| **Retention** | Flexible policies (daily, weekly, monthly, yearly) |
| **Geo-redundancy** | GRS/RA-GRS for backup data |

---

## 5.2.2 Recovery Services Vault

### What is a Recovery Services Vault?
- A **storage entity** in Azure that holds backup data, recovery points, and backup policies
- Required for Azure Backup and Azure Site Recovery
- Region-specific — create in the same region as resources (or paired region for DR)

### Key Properties

| Property | Detail |
|----------|--------|
| **Storage Redundancy** | LRS, ZRS, GRS (default) |
| **Soft Delete** | 14 days retention for deleted backup data (enabled by default) |
| **Cross-Region Restore** | Restore from secondary region (GRS vaults) |
| **Encryption** | Platform-managed keys (default) or customer-managed keys |
| **Immutability** | Prevent policy changes and deletion (for compliance) |

### Creating a Recovery Services Vault
```powershell
# Create vault
New-AzRecoveryServicesVault -ResourceGroupName "myRG" `
  -Name "myRecoveryVault" -Location "eastus"

# Set storage redundancy
$vault = Get-AzRecoveryServicesVault -Name "myRecoveryVault"
Set-AzRecoveryServicesBackupProperty -Vault $vault `
  -BackupStorageRedundancy GeoRedundant
```

```bash
# Create vault
az backup vault create --resource-group myRG \
  --name myRecoveryVault --location eastus

# Set storage redundancy (must be done before first backup)
az backup vault backup-properties set --resource-group myRG \
  --name myRecoveryVault --backup-storage-redundancy GeoRedundant
```

> [!WARNING]
> Storage redundancy for the vault must be set **before** the first backup is configured. Once a backup item is registered, the redundancy type **cannot be changed**.

## 5.2.3 Azure VM Backup

### How VM Backup Works
1. Azure Backup takes a **snapshot** of the VM (crash-consistent or app-consistent)
2. Snapshot is **transferred** to the Recovery Services vault
3. **Recovery points** are created based on the backup policy

### Backup Types

| Type | Description |
|------|-------------|
| **App-consistent** | Captures all data in memory + pending I/O (uses VSS on Windows, pre/post scripts on Linux) |
| **File-system consistent** | Consistent at file system level (Linux default) |
| **Crash-consistent** | Captures disk data only (as if VM crashed) — minimum guarantee |

### Backup Policy Settings

| Setting | Options |
|---------|---------|
| **Frequency** | Daily or hourly (Enhanced policy) |
| **Time** | Specific time of day |
| **Retention — Daily** | 7–9999 days |
| **Retention — Weekly** | 1–5163 weeks |
| **Retention — Monthly** | 1–1188 months |
| **Retention — Yearly** | 1–99 years |
| **Instant Restore** | 1–5 days (snapshot tier) |

### Configuring VM Backup — Portal
1. VM → Operations → Backup
2. Select/create **Recovery Services vault**
3. Select/create **backup policy**
4. Enable backup

### PowerShell Example
```powershell
# Get vault
$vault = Get-AzRecoveryServicesVault -Name "myRecoveryVault"
Set-AzRecoveryServicesVaultContext -Vault $vault

# Get backup policy
$policy = Get-AzRecoveryServicesBackupProtectionPolicy -Name "DefaultPolicy"

# Enable backup for VM
$vm = Get-AzVM -ResourceGroupName "myRG" -Name "myVM"
Enable-AzRecoveryServicesBackupProtection -ResourceGroupName "myRG" `
  -Name "myVM" -Policy $policy

# Trigger on-demand backup
$backupItem = Get-AzRecoveryServicesBackupItem -BackupManagementType AzureVM `
  -WorkloadType AzureVM -Name "myVM"
Backup-AzRecoveryServicesBackupItem -Item $backupItem
```

### Azure CLI Example
```bash
# Enable backup
az backup protection enable-for-vm --resource-group myRG \
  --vault-name myRecoveryVault --vm myVM \
  --policy-name DefaultPolicy

# Trigger on-demand backup
az backup protection backup-now --resource-group myRG \
  --vault-name myRecoveryVault --container-name myVM \
  --item-name myVM --retain-until 30-12-2024

# List recovery points
az backup recoverypoint list --resource-group myRG \
  --vault-name myRecoveryVault --container-name myVM \
  --item-name myVM --output table
```

### VM Restore Options

| Restore Type | Description |
|-------------|-------------|
| **Create a new VM** | Restore as a brand new VM |
| **Replace existing** | Replace disks of the existing VM |
| **Restore disks** | Restore only the managed disks (then attach manually) |
| **Cross-region restore** | Restore in the secondary paired region (GRS only) |
| **File recovery** | Mount recovery point as a drive to recover individual files |

### File Recovery (Individual File Restore)
1. Recovery Services vault → Backup Items → Select VM → File Recovery
2. Select a recovery point
3. Download and run the **script** on any machine
4. The recovery point mounts as a **drive** (iSCSI)
5. Copy the files you need
6. **Unmount** the drive when done

```bash
# Generate file recovery script
az backup restore files mount-rp --resource-group myRG \
  --vault-name myRecoveryVault --container-name myVM \
  --item-name myVM --rp-name <recovery-point>
```

> [!TIP]
> Use **File Recovery** when you only need to restore specific files instead of the entire VM. The recovery point is mounted as a drive on any Windows/Linux machine.

## 5.2.4 Azure File Share Backup

### How It Works
- Uses **share snapshots** (not agent-based)
- Backups stored in the Recovery Services vault
- Instantaneous backup and restore

### Key Features

| Feature | Detail |
|---------|--------|
| **Method** | Share snapshots |
| **Frequency** | Up to 4x daily (every 4, 6, 8, or 12 hours) |
| **Retention** | Daily (1–200), Weekly (1–200), Monthly (1–120), Yearly (1–10) |
| **Restore** | Full share or individual files/folders |
| **Max snapshots** | 200 per share (including backups) |

### Configuring File Share Backup
```bash
# Enable backup for file share
az backup protection enable-for-azurefileshare --resource-group myRG \
  --vault-name myRecoveryVault \
  --storage-account mystorageaccount \
  --azure-file-share myfileshare \
  --policy-name DailyFileSharePolicy
```

### Restore Options
- **Full share restore** — to original or alternate location
- **Item-level restore** — restore specific files/folders
- **Alternate location** — restore to a different storage account/file share

## 5.2.5 Azure SQL Database Backup

### Built-in Backup Features
- **Automatic backups** for all Azure SQL databases
- Uses **full, differential, and transaction log** backups
- **Point-in-time restore** to any second within the retention period

### Backup Schedule

| Backup Type | Frequency |
|-------------|-----------|
| **Full** | Weekly |
| **Differential** | Every 12–24 hours |
| **Transaction Log** | Every 5–10 minutes |

### Retention Periods

| Type | Retention |
|------|-----------|
| **PITR (Short-term)** | 1–35 days (default 7) |
| **Long-term retention (LTR)** | Weekly, Monthly, Yearly — up to 10 years |

### Point-in-Time Restore
```bash
# Restore to a point in time
az sql db restore --resource-group myRG \
  --server myserver --name mydb \
  --dest-name mydb-restored \
  --time "2024-01-15T10:30:00Z"
```

### Geo-Restore
- Restore from **geo-redundant backup** in the paired region
- Available even if the primary region is inaccessible
- RPO: Up to 1 hour (geo-replication lag)

### Long-Term Retention (LTR)
```bash
# Configure LTR policy
az sql db ltr-policy set --resource-group myRG \
  --server myserver --name mydb \
  --weekly-retention P4W \
  --monthly-retention P12M \
  --yearly-retention P5Y \
  --week-of-year 1
```

> [!NOTE]
> Azure SQL **automatic backups** are free (included in the service). You only pay for **long-term retention (LTR)** storage beyond the PITR retention period.

## 5.2.6 Azure Site Recovery (ASR)

### What is Azure Site Recovery?
- A **disaster recovery** service that replicates workloads to a secondary location
- Ensures business continuity during outages
- Supports Azure VMs, on-premises VMs (Hyper-V, VMware), and physical servers

### Key Concepts

| Concept | Description |
|---------|-------------|
| **Replication** | Continuous data replication to secondary region/site |
| **Failover** | Switch to the replica when primary fails |
| **Failback** | Return to the primary after recovery |
| **Recovery Plan** | Orchestrate failover steps for multi-tier applications |
| **RTO** | Recovery Time Objective — time to restore service |
| **RPO** | Recovery Point Objective — max acceptable data loss |

### ASR Scenarios

| Scenario | Source | Destination |
|----------|--------|-------------|
| **Azure to Azure** | Azure VMs in Region A | Azure VMs in Region B |
| **VMware to Azure** | On-premises VMware VMs | Azure |
| **Hyper-V to Azure** | On-premises Hyper-V VMs | Azure |
| **Physical to Azure** | Physical servers | Azure |

### Azure VM to Azure DR

| Feature | Detail |
|---------|--------|
| **Replication** | Continuous, asynchronous |
| **RPO** | Typically < 5 minutes |
| **RTO** | Minutes to hours (depends on data size) |
| **What replicates** | OS disk, data disks, VM configuration |
| **What doesn't replicate** | Temp disk, page file, cached data |

### Setting Up Azure-to-Azure DR
```bash
# Enable replication for Azure VM
az site-recovery protection-container mapping create \
  --resource-group myRG --vault-name myRecoveryVault \
  --fabric-name primaryFabric \
  --protection-container primaryContainer \
  --name myMapping \
  --policy-id <policy-id> \
  --target-protection-container-id <target-container-id>
```

### Recovery Plans
- Define the **order** in which VMs fail over
- Group VMs into **groups** that fail over together
- Add **manual actions** or **scripts** between groups
- **Test failover** without disrupting production

```
Recovery Plan: "MyAppDR"
├── Group 1: Database VMs (start first)
│   ├── SQLVM1
│   └── SQLVM2
├── Manual Action: Verify DB replication
├── Group 2: Application VMs
│   ├── AppVM1
│   └── AppVM2
└── Group 3: Web VMs (start last)
    ├── WebVM1
    └── WebVM2
```

### Test Failover
- **Validate** your DR plan without affecting production
- Creates VMs in an **isolated test network**
- Clean up test VMs after validation

### Failover Types

| Type | Description |
|------|-------------|
| **Test failover** | Non-disruptive validation; uses test network |
| **Planned failover** | Graceful failover; no data loss (Hyper-V) |
| **Failover** | Unplanned; triggered during actual outage |
| **Failback** | Return to primary after recovery |

> [!IMPORTANT]
> Always perform a **test failover** regularly to validate your DR plan. This does NOT affect production workloads and uses an isolated network.

## 5.2.7 Soft Delete for Backup Data

### What is Soft Delete?
- When backup data is deleted, it's **retained for 14 additional days**
- Prevents accidental or malicious deletion of backups
- **Enabled by default** on Recovery Services vaults

### Key Properties

| Property | Detail |
|----------|--------|
| **Retention** | 14 days after deletion |
| **State** | Soft-deleted items shown in vault |
| **Recovery** | Undelete within 14 days |
| **Billing** | No charge during soft delete period |
| **Scope** | VMs, SQL, SAP HANA, Azure Files |

### Multi-User Authorization (MUA)
- Requires **additional authorization** from a second user for critical operations
- Protects against: disable soft delete, change policy, stop backup with delete data
- Uses **Resource Guard** in a separate subscription

## 5.2.8 Azure Backup Center

### What is Backup Center?
- A **single unified management experience** for all backup activities
- Provides a **single pane of glass** across vaults, subscriptions, and regions
- Monitor, configure, and govern backup at scale

### Key Features

| Feature | Description |
|---------|-------------|
| **Dashboard** | Overview of all backup instances, jobs, alerts |
| **Configure** | Set up backup for new resources |
| **Restore** | Initiate restores from any vault |
| **Policies** | View and manage backup policies |
| **Reports** | Built-in reporting with Backup Reports |
| **Governance** | Azure Policy integration for backup compliance |

### Backup Reports
- Uses **Log Analytics** for detailed reporting
- Pre-built reports: usage, jobs, policies, optimization
- Track: backup storage consumption, cloud storage growth, retention trends

## Exam Tips — Section 5.2

- **Recovery Services vault** stores backup data; set redundancy (LRS/GRS) **before** first backup
- **VM backup types**: App-consistent (best) > File-system consistent > Crash-consistent
- **Instant restore**: Restore from snapshot tier (1–5 days) — fastest
- **File recovery**: Mount recovery point as a drive to restore individual files
- **VM restore options**: New VM, Replace existing, Restore disks, Cross-region, File recovery
- **Soft delete**: 14-day retention for deleted backups (enabled by default)
- **Azure File Share backup** uses share snapshots (max 200 per share)
- **Azure SQL auto-backup**: Full (weekly) + Differential (12–24h) + Log (5–10 min)
- **Point-in-time restore** for SQL: 1–35 days (default 7)
- **Long-term retention (LTR)**: Weekly, Monthly, Yearly — up to 10 years
- **Azure Site Recovery (ASR)** = disaster recovery; replicates VMs to secondary region
- ASR RPO: typically < 5 minutes; RTO: minutes to hours
- **Test failover** validates DR without affecting production
- **Recovery plans** define failover order for multi-tier applications
- **Backup Center** = single pane of glass for managing backups across subscriptions
- Know the difference: **Azure Backup** (data protection) vs **Azure Site Recovery** (disaster recovery)
- **Cross-region restore** requires GRS vault redundancy
