# 2.3 Configure Azure Files and Azure Blob Storage — AZ-104 Study Notes

---

## Overview

Azure provides two primary storage services for file and object data: **Azure Files** (managed file shares via SMB/NFS) and **Azure Blob Storage** (massively scalable object storage). Understanding when to use each and how to configure them is essential for the AZ-104 exam.

---

## 2.3.1 Azure Blob Storage

### What is Blob Storage?
- Microsoft's **object storage** solution for the cloud
- Optimized for storing **massive amounts of unstructured data** (text, binary, images, videos, logs)
- Accessible via REST API, Azure SDKs, PowerShell, CLI, or Azure Storage Explorer

### Blob Storage Hierarchy

```
Storage Account
└── Container (like a folder)
    ├── Blob 1 (file)
    ├── Blob 2
    └── Virtual Directory (using / in blob names)
        └── Blob 3
```

### Blob Types

| Type | Description | Max Size | Use Case |
|------|-------------|----------|----------|
| **Block Blob** | Composed of blocks; optimized for upload/download | ~190.7 TB | Files, images, videos, documents |
| **Append Blob** | Optimized for append operations | ~195 GB | Logging, audit trails |
| **Page Blob** | Random read/write; 512-byte pages | 8 TB | VHDs, VM disks |

### Container Access Levels

| Level | Description |
|-------|-------------|
| **Private** | No anonymous access (default) |
| **Blob** | Anonymous read access for blobs only |
| **Container** | Anonymous read access for container and blobs (list + read) |

> [!WARNING]
> Setting container access to **Blob** or **Container** allows anonymous public access. Microsoft recommends keeping containers **Private** and using SAS tokens or Microsoft Entra ID for access.

### PowerShell — Blob Operations
```powershell
$ctx = New-AzStorageContext -StorageAccountName "mystorageaccount" `
  -StorageAccountKey "<key>"

# Create container
New-AzStorageContainer -Name "mycontainer" -Context $ctx `
  -Permission Off   # Private

# Upload blob
Set-AzStorageBlobContent -Container "mycontainer" `
  -File "C:\data\myfile.txt" -Blob "uploads/myfile.txt" -Context $ctx

# Download blob
Get-AzStorageBlobContent -Container "mycontainer" `
  -Blob "uploads/myfile.txt" -Destination "C:\downloads\" -Context $ctx

# List blobs
Get-AzStorageBlob -Container "mycontainer" -Context $ctx

# Delete blob
Remove-AzStorageBlob -Container "mycontainer" -Blob "myfile.txt" -Context $ctx

# Copy blob
Start-AzStorageBlobCopy -SrcContainer "source" -SrcBlob "myfile.txt" `
  -DestContainer "destination" -DestBlob "myfile-copy.txt" -Context $ctx
```

### Azure CLI — Blob Operations
```bash
# Create container
az storage container create --name mycontainer \
  --account-name mystorageaccount --public-access off

# Upload blob
az storage blob upload --container-name mycontainer \
  --file "C:\data\myfile.txt" --name "uploads/myfile.txt" \
  --account-name mystorageaccount

# Download blob
az storage blob download --container-name mycontainer \
  --name "uploads/myfile.txt" --file "C:\downloads\myfile.txt" \
  --account-name mystorageaccount

# List blobs
az storage blob list --container-name mycontainer \
  --account-name mystorageaccount --output table
```

## 2.3.2 Configure Blob Storage Properties and Metadata

### System Properties
- Set by Azure automatically (e.g., `Last-Modified`, `ETag`, `Content-Type`, `Content-Length`)
- Some can be modified (e.g., `Content-Type`, `Cache-Control`)

### Custom Metadata
- **User-defined key-value pairs** on blobs, containers, and storage accounts
- Up to **8 KB** of metadata per blob
- Metadata names must be valid C# identifiers

```powershell
# Set blob metadata
$blob = Get-AzStorageBlob -Container "mycontainer" -Blob "myfile.txt" -Context $ctx
$blob.BlobClient.SetMetadata(@{ "Department" = "Finance"; "Project" = "Q4Report" })

# Set blob properties (content type)
$blob.BlobClient.SetHttpHeaders(
  [Azure.Storage.Blobs.Models.BlobHttpHeaders]::new() | ForEach-Object {
    $_.ContentType = "application/pdf"; $_
  }
)
```

## 2.3.3 Azure Files

### What is Azure Files?
- Fully managed **file shares** in the cloud
- Accessible via **SMB** (Server Message Block) and **NFS** (Network File System) protocols
- Can be mounted on Windows, Linux, and macOS

### Key Features

| Feature | Detail |
|---------|--------|
| **Protocols** | SMB 2.1, SMB 3.0, SMB 3.1.1, NFS 4.1 |
| **Max share size** | 100 TiB (with large file shares enabled) |
| **Max file size** | 4 TiB |
| **IOPS** | Up to 100,000 (Premium) or 20,000 (Standard w/ large shares) |
| **Encryption** | At rest (always on) + in transit (SMB 3.0+) |

### Azure Files vs. Azure Blob

| Feature | Azure Files | Azure Blob |
|---------|------------|------------|
| **Protocol** | SMB, NFS, REST | REST only |
| **Access pattern** | Mount as drive | API/SDK/tools |
| **Use case** | Lift-and-shift, shared config | Unstructured data, streaming |
| **Directory** | True directories | Virtual directories (/ in names) |
| **Locking** | File-level locks | Lease-based locking |

### Azure Files Tiers

| Tier | Description | Use Case |
|------|-------------|----------|
| **Premium** | SSD-backed, low latency | Databases, high IOPS workloads |
| **Transaction Optimized** | HDD-backed, optimized for transactions | Migration, dev/test |
| **Hot** | Optimized for general-purpose file sharing | Team shares, web content |
| **Cool** | Cost-optimized for infrequently accessed data | Backup, archival |

### Creating a File Share — Portal
1. Storage account → File shares → + File share
2. Enter **Name** and select **Tier**
3. Optionally set **quota** (max size in GiB)

### PowerShell — File Share Operations
```powershell
$ctx = New-AzStorageContext -StorageAccountName "mystorageaccount" `
  -StorageAccountKey "<key>"

# Create file share
New-AzStorageShare -Name "myfileshare" -Context $ctx

# Set quota
Set-AzStorageShareQuota -ShareName "myfileshare" -Quota 1024 -Context $ctx

# Create directory
New-AzStorageDirectory -ShareName "myfileshare" `
  -Path "documents" -Context $ctx

# Upload file
Set-AzStorageFileContent -ShareName "myfileshare" `
  -Source "C:\data\report.pdf" -Path "documents/report.pdf" -Context $ctx

# Download file
Get-AzStorageFileContent -ShareName "myfileshare" `
  -Path "documents/report.pdf" -Destination "C:\downloads\" -Context $ctx
```

### Azure CLI — File Share Operations
```bash
# Create file share
az storage share-rm create --storage-account mystorageaccount \
  --name myfileshare --quota 1024

# Upload file
az storage file upload --share-name myfileshare \
  --source "C:\data\report.pdf" --path "documents/report.pdf" \
  --account-name mystorageaccount

# List files
az storage file list --share-name myfileshare \
  --account-name mystorageaccount --output table
```

## 2.3.4 Mount Azure File Shares

### Mount on Windows
```cmd
# Map drive using net use
net use Z: \\mystorageaccount.file.core.windows.net\myfileshare /u:AZURE\mystorageaccount <storage-account-key>
```

### Mount on Linux
```bash
# Install cifs-utils
sudo apt install cifs-utils

# Create mount point
sudo mkdir -p /mnt/myfileshare

# Mount
sudo mount -t cifs //mystorageaccount.file.core.windows.net/myfileshare /mnt/myfileshare \
  -o vers=3.0,username=mystorageaccount,password=<key>,dir_mode=0777,file_mode=0777

# Persistent mount (add to /etc/fstab)
//mystorageaccount.file.core.windows.net/myfileshare /mnt/myfileshare cifs \
  vers=3.0,username=mystorageaccount,password=<key>,dir_mode=0777,file_mode=0777 0 0
```

### Port Requirements
- SMB requires **port 445** to be open
- Many ISPs block port 445 — use **VPN** or **Azure File Sync** as alternatives

> [!NOTE]
> If **port 445 is blocked** by your ISP, you can use: Azure VPN Gateway, ExpressRoute, Azure File Sync, or NFS (which uses port 2049).

## 2.3.5 Configure Azure File Sync

### What is Azure File Sync?
- **Centralizes** file shares in Azure Files while maintaining performance and compatibility of a Windows file server
- Transforms a Windows Server into a **quick cache** of your Azure file share
- Multi-site sync with cloud tiering

### Architecture Components

| Component | Description |
|-----------|-------------|
| **Storage Sync Service** | Top-level Azure resource for File Sync |
| **Sync Group** | Defines the sync topology; contains endpoints |
| **Cloud Endpoint** | The Azure file share |
| **Server Endpoint** | A path on a Windows Server |
| **Azure File Sync Agent** | Installed on Windows Server |
| **Registered Server** | A server trusted by the Storage Sync Service |

### Cloud Tiering
- **Automatically caches** frequently accessed files locally
- Less frequently accessed files are **tiered** (replaced with a pointer) to Azure
- Files are recalled seamlessly when accessed
- Policies: **Volume free space** policy and **Date** policy

### Setup Steps
1. Deploy **Storage Sync Service** in Azure
2. Install **Azure File Sync agent** on Windows Server
3. **Register** the server with Storage Sync Service
4. Create a **Sync Group** with a cloud endpoint (Azure file share)
5. Add **Server Endpoint** (folder on server)

### Key Requirements

| Requirement | Detail |
|-------------|--------|
| **OS** | Windows Server 2012 R2+ |
| **File system** | NTFS (ReFS not supported) |
| **Agent** | Azure File Sync agent required |
| **Network** | Port 443 outbound (HTTPS) |
| **Sync groups** | Max 100 sync groups per Storage Sync Service |
| **Server endpoints** | Max 1 per server per sync group |
| **Cloud endpoints** | 1 per sync group |

```powershell
# Register server (after agent install)
Register-AzStorageSyncServer -ResourceGroupName "myRG" `
  -StorageSyncServiceName "mySyncService"
```

## 2.3.6 Configure Blob Snapshots and Leases

### Blob Snapshots
- A **read-only copy** of a blob at a point in time
- Snapshot has the same name as base blob + a DateTime stamp
- Cannot modify a snapshot — only read, copy, or delete

```powershell
# Create snapshot
$blob = Get-AzStorageBlob -Container "mycontainer" -Blob "myfile.txt" -Context $ctx
$blob.BlobClient.CreateSnapshot()
```

### Blob Leases
- **Lock mechanism** to prevent other clients from modifying or deleting a blob
- Lease states: Available, Leased, Expired, Breaking, Broken

| Lease Duration | Detail |
|---------------|--------|
| **Fixed** | 15–60 seconds |
| **Infinite** | -1 (until explicitly released or broken) |

```powershell
# Acquire lease
$lease = $blob.BlobClient.GetBlobLeaseClient()
$lease.Acquire([TimeSpan]::FromSeconds(60))

# Release lease
$lease.Release()

# Break lease
$lease.Break()
```

## Exam Tips — Section 2.3

- **Block blobs** for most scenarios (files, images, videos); **Append blobs** for logs; **Page blobs** for VHDs
- Container access levels: **Private** (default, recommended), **Blob** (anonymous read to blobs), **Container** (anonymous read + list)
- **Azure Files** supports SMB and NFS protocols; requires **port 445** for SMB
- Azure Files tiers: **Premium** (SSD) > **Transaction Optimized** > **Hot** > **Cool**
- **Azure File Sync** caches Azure file shares on Windows Server with cloud tiering
- File Sync requires NTFS (not ReFS), Windows Server 2012 R2+, port 443 outbound
- **Cloud tiering** replaces less-used files with pointers; recalled on access
- Max file share size: **100 TiB** (large file shares enabled)
- Max individual file size: **4 TiB**
- **Snapshots** are read-only point-in-time copies; **Leases** provide locking
- Know the blob metadata limit: **8 KB per blob**
- **AzCopy** and **Storage Explorer** are key tools for managing blob data
- NFS shares require **Premium file shares** and a VNet with a service or private endpoint
