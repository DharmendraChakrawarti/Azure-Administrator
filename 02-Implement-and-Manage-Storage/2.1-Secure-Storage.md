# 2.1 Secure Storage — AZ-104 Study Notes

---

## Overview

Securing Azure Storage involves controlling **who can access** your data, **how** they access it, and ensuring data is **encrypted** both in transit and at rest. Azure provides multiple layers of security including access keys, shared access signatures (SAS), Microsoft Entra ID authorization, encryption, and network rules.

> [!IMPORTANT]
> Storage security is a heavily tested area on AZ-104. Know the differences between access keys, SAS tokens, stored access policies, and Microsoft Entra authorization.

---

## 2.1.1 Configure Azure Storage Firewalls and Virtual Networks

### Network Access Rules

By default, storage accounts accept connections from **all networks**. You can restrict this to specific networks.

### Configuration Options

| Setting | Description |
|---------|-------------|
| **All networks** | Default — open to the internet |
| **Selected networks** | Restrict to specific VNets, subnets, and IP ranges |
| **Disabled** | No public network access |

### Virtual Network Rules
- Allow access from specific **VNet subnets** using **service endpoints**
- Requires enabling **Microsoft.Storage** service endpoint on the subnet
- Up to **200** VNet rules per storage account

### IP Network Rules
- Allow access from specific **public IP addresses or CIDR ranges**
- Up to **200** IP rules per storage account
- Private IP addresses (RFC 1918) are **NOT allowed** in IP rules

### PowerShell Examples
```powershell
# Set default action to deny
Update-AzStorageAccountNetworkRuleSet -ResourceGroupName "myRG" `
  -Name "mystorageaccount" -DefaultAction Deny

# Add VNet rule
$subnet = Get-AzVirtualNetwork -ResourceGroupName "myRG" -Name "myVNet" |
  Get-AzVirtualNetworkSubnetConfig -Name "mySubnet"
Add-AzStorageAccountNetworkRule -ResourceGroupName "myRG" `
  -Name "mystorageaccount" -VirtualNetworkResourceId $subnet.Id

# Add IP rule
Add-AzStorageAccountNetworkRule -ResourceGroupName "myRG" `
  -Name "mystorageaccount" -IPAddressOrRange "203.0.113.0/24"
```

### Azure CLI Examples
```bash
# Set default action to deny
az storage account update --name mystorageaccount \
  --resource-group myRG --default-action Deny

# Add VNet rule
az storage account network-rule add --account-name mystorageaccount \
  --resource-group myRG --vnet-name myVNet --subnet mySubnet

# Add IP rule
az storage account network-rule add --account-name mystorageaccount \
  --resource-group myRG --ip-address 203.0.113.0/24
```

### Trusted Azure Services
- Some Azure services operate from IPs that can't be added to network rules
- Enable **"Allow trusted Microsoft services to access this storage account"** exception
- Trusted services include: Azure Backup, Azure Site Recovery, Azure DevTest Labs, Azure Event Grid, Azure Event Hubs, Azure Monitor, Azure SQL Data Warehouse, etc.

> [!TIP]
> When you set the default action to **Deny**, you must also add a network rule or exception for your own access, otherwise you will lock yourself out.

## 2.1.2 Configure Storage Account Access Keys

### What Are Access Keys?
- **Two 512-bit keys** (key1, key2) generated when a storage account is created
- Provide **full access** to the storage account — equivalent to root password
- Used to authorize access via **Shared Key** authorization

### Key Properties

| Property | Detail |
|----------|--------|
| **Number of keys** | 2 (key1 and key2) |
| **Permissions** | Full access to all data and configuration |
| **Rotation** | Rotate regularly; use two keys for zero-downtime rotation |
| **Best practice** | Use Microsoft Entra ID instead when possible |

### Key Rotation Strategy
1. Update connection strings in apps to use **key2**
2. Regenerate **key1** in Azure portal
3. Update connection strings to use **new key1**
4. Regenerate **key2**

### PowerShell Examples
```powershell
# Get storage account keys
Get-AzStorageAccountKey -ResourceGroupName "myRG" -Name "mystorageaccount"

# Regenerate key1
New-AzStorageAccountKey -ResourceGroupName "myRG" `
  -Name "mystorageaccount" -KeyName "key1"
```

### Azure CLI
```bash
# List keys
az storage account keys list --account-name mystorageaccount --resource-group myRG

# Regenerate key1
az storage account keys renew --account-name mystorageaccount \
  --resource-group myRG --key key1
```

> [!WARNING]
> Access keys grant **unrestricted access** to the entire storage account. Anyone with the key can read, write, and delete any data. Protect them like passwords.

## 2.1.3 Configure Shared Access Signatures (SAS)

### What is a SAS?
- A **URI** that grants restricted access to Azure Storage resources
- Provides **granular control** over permissions, resources, time, and IP
- Does NOT require sharing account keys

### SAS Types

| Type | Signed With | Scope |
|------|------------|-------|
| **User Delegation SAS** | Microsoft Entra credentials (user delegation key) | Blob storage only |
| **Service SAS** | Storage account key | Single service (Blob, Queue, Table, or File) |
| **Account SAS** | Storage account key | One or more services + service-level operations |

### SAS Components (URI Parameters)

| Parameter | Description | Example |
|-----------|-------------|---------|
| `sv` | Storage service version | `2022-11-02` |
| `ss` | Services (b=blob, f=file, q=queue, t=table) | `bfqt` |
| `srt` | Resource types (s=service, c=container, o=object) | `sco` |
| `sp` | Permissions (r=read, w=write, d=delete, l=list, a=add, c=create, u=update, p=process) | `rwdlac` |
| `se` | Expiry time (UTC) | `2024-12-31T23:59:59Z` |
| `st` | Start time (UTC) | `2024-01-01T00:00:00Z` |
| `sip` | Allowed IP range | `203.0.113.0-203.0.113.255` |
| `spr` | Protocol (https, https,http) | `https` |
| `sig` | Signature (HMAC-SHA256) | `<hash>` |

### Creating a SAS — Portal
Storage account → Shared access signature → Configure permissions/expiry → Generate SAS

### PowerShell Example
```powershell
$ctx = New-AzStorageContext -StorageAccountName "mystorageaccount" `
  -StorageAccountKey "<key>"

# Account SAS
$sas = New-AzStorageAccountSASToken -Service Blob,File,Table,Queue `
  -ResourceType Service,Container,Object `
  -Permission "rwdlacup" `
  -ExpiryTime (Get-Date).AddHours(2) `
  -Context $ctx

# Container SAS
$containerSas = New-AzStorageContainerSASToken -Name "mycontainer" `
  -Permission "rl" -ExpiryTime (Get-Date).AddDays(7) -Context $ctx

# Blob SAS
$blobSas = New-AzStorageBlobSASToken -Container "mycontainer" `
  -Blob "myfile.txt" -Permission "r" -ExpiryTime (Get-Date).AddHours(1) `
  -Context $ctx
```

### Best Practices for SAS
1. Always use **HTTPS** (`spr=https`)
2. Set the **shortest practical expiry** time
3. Grant **minimum required permissions**
4. Use **User Delegation SAS** when possible (most secure)
5. Use **Stored Access Policies** for Service SAS (allows revocation)
6. Have a **revocation plan** — regenerate keys to invalidate SAS tokens

### User Delegation SAS (Recommended)

| Feature | Detail |
|---------|--------|
| **Security** | Most secure — uses Microsoft Entra credentials |
| **Scope** | Blob storage only |
| **Key source** | User delegation key (not account key) |
| **Revocation** | Revoke user delegation key to invalidate |
| **Max validity** | 7 days from creation |

```powershell
# Create user delegation SAS (requires Microsoft Entra auth)
$ctx = New-AzStorageContext -StorageAccountName "mystorageaccount" -UseConnectedAccount
$startTime = Get-Date
$endTime = $startTime.AddDays(1)
$userDelegationKey = New-AzStorageUserDelegationKey -StartTime $startTime `
  -ExpiryTime $endTime -Context $ctx
```

> [!IMPORTANT]
> **User Delegation SAS** is the most secure SAS type because it uses Microsoft Entra ID credentials instead of the account key. Microsoft recommends using it whenever possible.

## 2.1.4 Configure Stored Access Policies

### What Are Stored Access Policies?
- Define **reusable permission sets** on a storage container, queue, table, or file share
- A SAS token linked to a stored access policy **can be revoked** by modifying or deleting the policy
- Provides an additional level of control over Service SAS tokens

### Key Properties

| Property | Detail |
|----------|--------|
| **Max per container** | 5 stored access policies |
| **Parameters** | Start time, expiry time, permissions |
| **Revocation** | Change start time to past, remove permissions, or delete the policy |
| **Scope** | Container, queue, table, or file share level |

### PowerShell Example
```powershell
$ctx = New-AzStorageContext -StorageAccountName "mystorageaccount" `
  -StorageAccountKey "<key>"

# Create stored access policy
New-AzStorageContainerStoredAccessPolicy -Container "mycontainer" `
  -Policy "ReadOnlyPolicy" -Permission "rl" `
  -ExpiryTime (Get-Date).AddMonths(6) -Context $ctx

# Generate SAS from stored access policy
$sas = New-AzStorageContainerSASToken -Name "mycontainer" `
  -Policy "ReadOnlyPolicy" -Context $ctx

# Revoke: update policy with past start time
Set-AzStorageContainerStoredAccessPolicy -Container "mycontainer" `
  -Policy "ReadOnlyPolicy" -StartTime (Get-Date).AddDays(-1) `
  -ExpiryTime (Get-Date).AddDays(-1) -Context $ctx
```

> [!TIP]
> Unlike ad-hoc SAS tokens, SAS tokens linked to a **stored access policy** can be **revoked** without regenerating the storage account key. This makes them the preferred approach for Service SAS.

## 2.1.5 Configure Microsoft Entra ID Authorization for Storage

### Authorization Methods Comparison

| Method | Security Level | Scope |
|--------|---------------|-------|
| **Microsoft Entra ID** | ★★★★★ Highest | Blob and Queue |
| **User Delegation SAS** | ★★★★ High | Blob only |
| **Stored Access Policy + SAS** | ★★★ Medium | Container/Queue/Table/File |
| **Service/Account SAS** | ★★ Low | Per token |
| **Shared Key (Access Keys)** | ★ Lowest | Full account |

### Azure RBAC Data Roles for Storage

| Role | Blob | Queue | Description |
|------|------|-------|-------------|
| **Storage Blob Data Reader** | ✅ | — | Read blobs and list containers |
| **Storage Blob Data Contributor** | ✅ | — | Read, write, delete blobs |
| **Storage Blob Data Owner** | ✅ | — | Full access + POSIX ACL for Data Lake |
| **Storage Queue Data Reader** | — | ✅ | Read and peek queue messages |
| **Storage Queue Data Contributor** | — | ✅ | Read, add, update, delete messages |
| **Storage Queue Data Message Processor** | — | ✅ | Peek, get, delete queue messages |
| **Storage Queue Data Message Sender** | — | ✅ | Add messages to queue |

### Disabling Shared Key Authorization
```powershell
# Force Microsoft Entra-only access
Set-AzStorageAccount -ResourceGroupName "myRG" `
  -Name "mystorageaccount" -AllowSharedKeyAccess $false
```
```bash
az storage account update --name mystorageaccount \
  --resource-group myRG --allow-shared-key-access false
```

> [!NOTE]
> Disabling Shared Key access forces all requests to use **Microsoft Entra authorization**. This also breaks SAS tokens (which are derived from the account key).

## 2.1.6 Configure Storage Encryption

### Encryption at Rest

| Feature | Detail |
|---------|--------|
| **Default** | All data encrypted with **256-bit AES** (FIPS 140-2 compliant) |
| **Automatic** | Enabled by default; cannot be disabled |
| **Scope** | All services (Blob, File, Queue, Table) and all tiers |
| **Key management** | Microsoft-managed keys (default) or customer-managed keys (CMK) |

### Key Management Options

| Option | Stored In | Customer Control |
|--------|-----------|-----------------|
| **Microsoft-managed keys** | Microsoft managed | None (fully managed) |
| **Customer-managed keys (CMK)** | Azure Key Vault | Customer rotates keys |
| **Customer-provided keys** | Customer's infrastructure | Per-request key in header |

### Customer-Managed Keys (CMK)
- Stored in **Azure Key Vault** or **Azure Key Vault Managed HSM**
- Supported for Blob and File services
- Storage account must be in the **same region** as the Key Vault
- Requires a **managed identity** (system-assigned or user-assigned) to access the Key Vault

```powershell
# Configure CMK
Set-AzStorageAccount -ResourceGroupName "myRG" `
  -Name "mystorageaccount" `
  -KeyvaultEncryption `
  -KeyName "myEncryptionKey" `
  -KeyVaultUri "https://mykeyvault.vault.azure.net/"
```

### Encryption in Transit
- **HTTPS** enforced by default (Secure transfer required = Enabled)
- **SMB 3.0** encryption for Azure Files

```powershell
# Ensure HTTPS is required
Set-AzStorageAccount -ResourceGroupName "myRG" `
  -Name "mystorageaccount" -EnableHttpsTrafficOnly $true
```

### Infrastructure Encryption (Double Encryption)
- Adds a **second layer** of encryption at the infrastructure level
- Uses a **different encryption algorithm/key** than service-level encryption
- Must be enabled **at account creation** — cannot be enabled later

> [!CAUTION]
> **Infrastructure encryption** (double encryption) can only be enabled when the storage account is created. It cannot be added after the fact.

## 2.1.7 Configure Private Endpoints for Storage

### What Are Private Endpoints?
- A **network interface** with a private IP in your VNet
- Routes traffic to your storage account **over the Microsoft backbone network**
- Eliminates exposure to the public internet

### Key Properties

| Property | Detail |
|----------|--------|
| **IP address** | Private IP from your VNet/subnet |
| **DNS** | Private DNS zone maps `storageaccount.blob.core.windows.net` to private IP |
| **Network traffic** | Stays on Microsoft backbone — never traverses the internet |
| **Multiple endpoints** | One per storage sub-resource (blob, file, queue, table, web, dfs) |

### Creating a Private Endpoint — Portal
1. Storage account → Networking → Private endpoint connections → + Private endpoint
2. Select subscription, resource group, name, region
3. Select target sub-resource (blob, file, etc.)
4. Select VNet and subnet
5. Integrate with private DNS zone

### PowerShell Example
```powershell
$privateEndpointConn = New-AzPrivateLinkServiceConnection `
  -Name "myStoragePEConnection" `
  -PrivateLinkServiceId $storageAccount.Id `
  -GroupId "blob"

$privateEndpoint = New-AzPrivateEndpoint `
  -ResourceGroupName "myRG" `
  -Name "myStoragePE" `
  -Location "eastus" `
  -Subnet $subnet `
  -PrivateLinkServiceConnection $privateEndpointConn
```

> [!TIP]
> When using private endpoints, you should also set the public network access to **Disabled** and configure **Private DNS Zones** for seamless name resolution.

## Exam Tips — Section 2.1

- **Access keys** = full unrestricted access, rotate regularly using two-key strategy
- **SAS types**: User Delegation (most secure, Blob only) > Service SAS > Account SAS
- **User Delegation SAS** max validity = **7 days**
- **Stored access policies**: max 5 per container; provide SAS revocation
- **Microsoft Entra ID** authorization is the most secure method overall
- **Storage Account Contributor** ≠ data access (control plane only)
- **Storage Blob Data *** roles = data plane access
- **Encryption at rest** is always on, AES-256, cannot be disabled
- **Customer-managed keys (CMK)** stored in Azure Key Vault
- **Infrastructure encryption** (double encryption) must be set at **account creation**
- **Secure transfer required** enforces HTTPS
- **Private endpoints** use private IPs and keep traffic on Microsoft backbone
- **Network rules** default action: Allow → change to **Deny** and add exceptions
- Know the difference: **service endpoints** (extend VNet identity) vs. **private endpoints** (private IP in VNet)
