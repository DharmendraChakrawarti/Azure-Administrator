# 2.2 Manage Storage — AZ-104 Study Notes

---

## Overview

Managing Azure Storage involves creating and configuring storage accounts, managing data access and replication, and understanding the different storage services (Blob, File, Queue, Table). This section covers storage account creation, configuration, data import/export, and replication strategies.

---

## 2.2.1 Create and Configure Storage Accounts

### What is an Azure Storage Account?
- A **unique namespace** in Azure for your data
- Accessible from anywhere via HTTP/HTTPS
- Contains all Azure Storage data objects: blobs, files, queues, tables, disks

### Storage Account Types

| Type | Supported Services | Performance | Use Case |
|------|-------------------|-------------|----------|
| **Standard general-purpose v2** | Blob, File, Queue, Table, Data Lake | Standard (HDD) | Most scenarios (recommended) |
| **Premium block blobs** | Blob (block blobs + append blobs) | Premium (SSD) | High transaction rates, low latency |
| **Premium file shares** | File shares only | Premium (SSD) | Enterprise file shares, high IOPS |
| **Premium page blobs** | Page blobs only | Premium (SSD) | VM disks (unmanaged) |

> [!IMPORTANT]
> **General-purpose v2** is the recommended account type for most scenarios. Premium accounts provide SSD-backed storage for specific high-performance workloads.

### Creating a Storage Account — Portal
1. Storage accounts → + Create
2. **Basics**: Subscription, Resource Group, Name (globally unique, 3–24 chars, lowercase + numbers), Region, Performance (Standard/Premium), Redundancy
3. **Advanced**: Security, Data Lake Gen2, Blob access tier, TLS version
4. **Networking**: Connectivity, Network routing
5. **Data Protection**: Recovery, Tracking, Access control
6. **Encryption**: Encryption type, Key management
7. **Tags**: Key-value pairs
8. **Review + Create**

### Storage Account Name Rules

| Rule | Detail |
|------|--------|
| **Length** | 3–24 characters |
| **Characters** | Lowercase letters and numbers only |
| **Uniqueness** | Globally unique across Azure |

### PowerShell Example
```powershell
# Create storage account
New-AzStorageAccount -ResourceGroupName "myRG" `
  -Name "mystorageaccount" `
  -Location "eastus" `
  -SkuName "Standard_LRS" `
  -Kind "StorageV2" `
  -AccessTier "Hot" `
  -MinimumTlsVersion "TLS1_2"
```

### Azure CLI Example
```bash
az storage account create --name mystorageaccount \
  --resource-group myRG --location eastus \
  --sku Standard_LRS --kind StorageV2 \
  --access-tier Hot --min-tls-version TLS1_2
```

## 2.2.2 Configure Storage Account Redundancy/Replication

### Redundancy Options

| Option | Full Name | Copies | Locations | Durability (nines) |
|--------|-----------|--------|-----------|-------------------|
| **LRS** | Locally Redundant Storage | 3 | Single datacenter | 11 nines |
| **ZRS** | Zone-Redundant Storage | 3 | 3 availability zones | 12 nines |
| **GRS** | Geo-Redundant Storage | 6 | Primary + secondary region | 16 nines |
| **GZRS** | Geo-Zone-Redundant Storage | 6 | 3 AZs in primary + secondary region | 16 nines |
| **RA-GRS** | Read-Access GRS | 6 | GRS + read access to secondary | 16 nines |
| **RA-GZRS** | Read-Access GZRS | 6 | GZRS + read access to secondary | 16 nines |

### Understanding Redundancy

```
LRS:     [DC1: Copy1, Copy2, Copy3]
ZRS:     [Zone1: Copy1] [Zone2: Copy2] [Zone3: Copy3]
GRS:     [Primary DC: 3 copies] → async → [Secondary DC: 3 copies]
GZRS:    [Primary: 3 AZ copies] → async → [Secondary DC: 3 copies]
RA-GRS:  GRS + read access to secondary endpoint
RA-GZRS: GZRS + read access to secondary endpoint
```

### Key Facts

| Feature | LRS | ZRS | GRS | GZRS |
|---------|-----|-----|-----|------|
| **Node failures** | ✅ | ✅ | ✅ | ✅ |
| **Zone failures** | ❌ | ✅ | ❌* | ✅ |
| **Region failures** | ❌ | ❌ | ✅ | ✅ |
| **Read from secondary** | ❌ | ❌ | RA-GRS only | RA-GZRS only |
| **Cost** | Lowest | Medium | Medium-High | Highest |

*GRS protects against region failure in the primary, but data is LRS in both regions.

### Failover for Geo-Redundant Accounts
- **Automatic failover**: Microsoft-initiated during major regional outage
- **Customer-initiated failover**: You trigger failover manually
- After failover: secondary becomes primary, replication becomes LRS
- RPO: Typically < 15 minutes (async replication)

```powershell
# Change redundancy
Set-AzStorageAccount -ResourceGroupName "myRG" `
  -Name "mystorageaccount" -SkuName "Standard_GRS"

# Initiate failover
Invoke-AzStorageAccountFailover -ResourceGroupName "myRG" `
  -Name "mystorageaccount"
```

> [!WARNING]
> After a failover, the storage account becomes **LRS** in the new primary region. You must manually reconfigure geo-redundancy. There is **potential data loss** due to the async nature of replication.

## 2.2.3 Configure Object Replication

### What is Object Replication?
- **Asynchronously copies blobs** between a source and destination storage account
- Works across different regions, subscriptions, or Azure AD tenants
- Requires **blob versioning** enabled on both accounts and **change feed** on the source

### Key Requirements

| Requirement | Detail |
|-------------|--------|
| **Blob versioning** | Enabled on both source and destination |
| **Change feed** | Enabled on source account |
| **Account type** | General-purpose v2 or Premium block blob |
| **Blob types** | Block blobs only (not page or append blobs) |
| **Snapshots** | NOT replicated |
| **Blob tier** | Source and destination can be different tiers |

### Use Cases
- **Latency reduction** — replicate data closer to consumers
- **Compliance** — maintain copies in specific regions
- **Data analytics** — replicate to a separate account for analysis

### PowerShell Example
```powershell
# Create replication policy
$rule = New-AzStorageObjectReplicationPolicyRule `
  -SourceContainer "source-container" `
  -DestinationContainer "dest-container"

Set-AzStorageObjectReplicationPolicy -ResourceGroupName "myRG" `
  -StorageAccountName "destAccount" `
  -SourceAccount "sourceAccount" `
  -Rule $rule
```

## 2.2.4 Configure Storage Account Access Tiers

### Blob Access Tiers

| Tier | Description | Storage Cost | Access Cost | Min Duration |
|------|-------------|-------------|-------------|-------------|
| **Hot** | Frequently accessed data | Highest | Lowest | None |
| **Cool** | Infrequently accessed; stored ≥30 days | Lower | Higher | 30 days |
| **Cold** | Rarely accessed; stored ≥90 days | Lower still | Higher still | 90 days |
| **Archive** | Rarely accessed; stored ≥180 days | Lowest | Highest | 180 days |

### Tier Configuration Levels
1. **Account level** — Set default tier (Hot or Cool only)
2. **Blob level** — Override per blob (Hot, Cool, Cold, or Archive)

### Archive Tier Special Rules
- Data is **offline** — cannot be read or modified directly
- Must **rehydrate** to Hot or Cool before accessing
- Rehydration options:
  - **Standard priority**: Up to 15 hours
  - **High priority**: Under 1 hour for blobs < 10 GB

```powershell
# Set account default tier
Set-AzStorageAccount -ResourceGroupName "myRG" `
  -Name "mystorageaccount" -AccessTier "Cool"

# Set blob tier
$ctx = New-AzStorageContext -StorageAccountName "mystorageaccount" -StorageAccountKey "<key>"
Set-AzStorageBlobTier -Container "mycontainer" -Blob "myblob.txt" `
  -Tier "Archive" -Context $ctx

# Rehydrate from archive
Set-AzStorageBlobTier -Container "mycontainer" -Blob "myblob.txt" `
  -Tier "Hot" -RehydratePriority High -Context $ctx
```

### Lifecycle Management Policies
- Automate **tier transitions** and **deletion** based on rules
- Rules based on: last modified date, creation date, last accessed date

```json
{
  "rules": [
    {
      "name": "MoveToCool",
      "type": "Lifecycle",
      "definition": {
        "actions": {
          "baseBlob": {
            "tierToCool": { "daysAfterModificationGreaterThan": 30 },
            "tierToArchive": { "daysAfterModificationGreaterThan": 180 },
            "delete": { "daysAfterModificationGreaterThan": 365 }
          },
          "snapshot": {
            "delete": { "daysAfterCreationGreaterThan": 90 }
          }
        },
        "filters": {
          "blobTypes": ["blockBlob"],
          "prefixMatch": ["logs/"]
        }
      }
    }
  ]
}
```

> [!TIP]
> Use **lifecycle management policies** to automatically move blobs to cooler tiers and reduce costs. This is an important exam topic — know the minimum storage durations for each tier.

## 2.2.5 Import and Export Data to Azure

### Azure Import/Export Service
- Physically ship **disk drives** to Azure datacenters for large data transfers
- Supports: Azure Blob Storage and Azure Files

| Direction | Supported Storage |
|-----------|------------------|
| **Import** | Azure Blob Storage, Azure Files |
| **Export** | Azure Blob Storage only |

### Process
1. Create an Import/Export job in Azure portal
2. Prepare drives using the **WAImportExport** tool
3. Ship drives to Azure datacenter
4. Azure copies data and ships drives back

### AzCopy
- **Command-line tool** for copying data to/from Azure Storage
- Supports: Blob, File, Table
- **Incremental copy** and **synchronization** support
- Authentication via SAS token or Microsoft Entra ID

```bash
# Upload files to blob
azcopy copy "C:\data\*" "https://mystorageaccount.blob.core.windows.net/mycontainer?<SAS>"

# Download blobs
azcopy copy "https://mystorageaccount.blob.core.windows.net/mycontainer/*?<SAS>" "C:\data\"

# Sync (only copy changed/new files)
azcopy sync "C:\data" "https://mystorageaccount.blob.core.windows.net/mycontainer?<SAS>"

# Copy between storage accounts
azcopy copy "https://source.blob.core.windows.net/container?<SAS>" \
  "https://dest.blob.core.windows.net/container?<SAS>" --recursive
```

### Azure Storage Explorer
- **GUI tool** for managing Azure Storage (free, cross-platform)
- Connect with: Microsoft Entra ID, connection string, SAS token, storage account key
- Supports: Blob, File, Queue, Table, Data Lake

### Azure Data Box Family

| Product | Capacity | Use Case |
|---------|----------|----------|
| **Data Box Disk** | Up to 35 TB (5 × 8 TB disks) | Small-medium transfers |
| **Data Box** | 80 TB | Large transfers |
| **Data Box Heavy** | 800 TB | Very large transfers |

## 2.2.6 Configure Soft Delete and Versioning

### Blob Soft Delete
- Retains deleted blobs for a specified **retention period** (1–365 days)
- Deleted blobs can be **restored** during retention period
- Default: **7 days** retention

### Container Soft Delete
- Retains deleted containers for **1–365 days**
- Restore deleted containers via Azure portal or API

### Blob Versioning
- Automatically maintains **previous versions** of a blob when modified/overwritten
- Each version has a unique **version ID**
- Versions can be listed, downloaded, or promoted to current

```powershell
# Enable blob soft delete
Enable-AzStorageBlobDeleteRetentionPolicy -ResourceGroupName "myRG" `
  -StorageAccountName "mystorageaccount" -RetentionDays 14

# Enable container soft delete
Enable-AzStorageContainerDeleteRetentionPolicy -ResourceGroupName "myRG" `
  -StorageAccountName "mystorageaccount" -RetentionDays 7

# Enable versioning
Update-AzStorageBlobServiceProperty -ResourceGroupName "myRG" `
  -StorageAccountName "mystorageaccount" -IsVersioningEnabled $true
```

### Point-in-Time Restore for Block Blobs
- Restore block blobs to a **previous state** at a specific point in time
- Requires: Soft delete, Change feed, Blob versioning all enabled
- Restore scope: Entire account, specific containers, or blob prefixes

> [!NOTE]
> Blob versioning, soft delete, and change feed must ALL be enabled to use **point-in-time restore**. This is a common exam question.

## Exam Tips — Section 2.2

- **Storage account name**: 3–24 characters, lowercase + numbers, globally unique
- **GPv2** is the recommended account type for most scenarios
- Know redundancy options: **LRS** (cheapest) → **ZRS** → **GRS** → **RA-GZRS** (most durable + readable)
- **RA-GRS/RA-GZRS** = secondary endpoint is readable; access via `*-secondary.blob.core.windows.net`
- After geo-failover → account becomes **LRS**
- **Object replication** requires blob versioning (both) + change feed (source)
- Access tiers: **Hot** > **Cool** (30d) > **Cold** (90d) > **Archive** (180d, offline)
- **Archive tier** is offline — must rehydrate before access (standard: ≤15h, high priority: <1h)
- **Lifecycle management** automates tier transitions — know the JSON rule structure
- **AzCopy** is the CLI tool for data transfer; supports sync and Microsoft Entra auth
- **Data Box** for large offline data transfers (80 TB)
- **Soft delete** retention: 1–365 days; default 7 days
- **Point-in-time restore** requires: soft delete + change feed + blob versioning
