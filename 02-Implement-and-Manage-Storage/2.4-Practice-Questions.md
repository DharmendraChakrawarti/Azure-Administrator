# 2.4 Practice Questions — Implement and Manage Storage

> Scenario-based questions to test your understanding. Answers are at the bottom.

---

## Questions

### Q1 — SAS Token Types
Your application needs temporary read-only access to blobs in a storage account. You want the most secure SAS type possible and the ability to revoke access. Which SAS type should you use, and what is the maximum validity period?

---

### Q2 — Storage Redundancy
A company requires that their storage data survives a complete regional outage AND that the secondary region data is readable at all times for reporting purposes. Which redundancy option should they choose?

---

### Q3 — Access Tiers
You have 5 TB of log data that is accessed frequently for the first 30 days, rarely accessed for the next 150 days, and never accessed after that but must be kept for 2 years for compliance. Design a lifecycle management strategy.

---

### Q4 — Storage Firewalls
After configuring a storage account firewall with default action set to **Deny**, your application running on an Azure VM in a VNet can no longer access the storage account. What two options do you have to restore access?

---

### Q5 — Encryption
Your security team requires that all storage data be encrypted with keys that your organization manages and rotates. They also want a second layer of encryption at the infrastructure level. What should you configure?

---

### Q6 — Azure File Sync
Your company has three branch offices, each with a Windows file server. Users at all locations need access to the same set of files, but with fast local performance. Local storage is limited, so only frequently used files should be kept locally. What should you configure?

---

### Q7 — Blob Versioning and Point-in-Time Restore
You need to enable point-in-time restore for block blobs in a storage account. What three features must be enabled before you can configure point-in-time restore?

---

### Q8 — Storage Account Access Keys
A developer accidentally committed a storage account access key to a public GitHub repository. What should you do immediately, and what should you implement to prevent this in the future?

---

### Q9 — Stored Access Policy
You generated 50 SAS tokens linked to a stored access policy on a container. You now need to immediately revoke all 50 tokens. How can you do this without regenerating the storage account key?

---

### Q10 — Object Replication
You want to replicate blobs from a storage account in East US to a storage account in West Europe for latency reduction. What prerequisites must be met on both storage accounts?

---

### Q11 — Archive Tier
A user tries to read a blob in the Archive tier and gets an error. Why? What must they do to access the data, and how long will it take?

---

### Q12 — Azure Files vs Blob Storage
Your team is debating whether to use Azure Files or Azure Blob Storage for a lift-and-shift migration of an on-premises application that reads/writes files from a mounted network drive (UNC path). Which should you recommend and why?

---

### Q13 — Storage Account Name
A developer tries to create a storage account named "My-Storage-Account" and gets an error. Another developer tries "mystorageaccount123!" and also gets an error. What are the naming rules?

---

### Q14 — Private Endpoints
After configuring a private endpoint for blob storage, users report that they can still access the storage account from the public internet. What additional step is needed?

---

### Q15 — Soft Delete
A team member accidentally deleted a blob container with important data. The container was deleted 10 days ago. Container soft delete is enabled with 14-day retention. Can the container be recovered? What if it was deleted 20 days ago?

---

## Answers

### A1
Use a **User Delegation SAS** — it's the most secure type because it uses Microsoft Entra ID credentials instead of the storage account key. Maximum validity is **7 days**. To enable revocation, you can revoke the **user delegation key**. Note: User Delegation SAS works only with **Blob storage**.

### A2
**RA-GRS (Read-Access Geo-Redundant Storage)** or **RA-GZRS** (if zone redundancy in primary is also needed). RA-GRS stores 3 copies in the primary region + 3 copies in the secondary region, AND allows read access to the secondary endpoint (`*-secondary.blob.core.windows.net`).

### A3
Create a **lifecycle management policy**:
- **Days 0–30**: Keep in **Hot** tier (frequently accessed)
- **After 30 days**: Move to **Cool** tier (`tierToCool: daysAfterModificationGreaterThan: 30`)
- **After 180 days**: Move to **Archive** tier (`tierToArchive: daysAfterModificationGreaterThan: 180`)
- **After 730 days** (2 years): Delete (`delete: daysAfterModificationGreaterThan: 730`)

### A4
Two options:
1. **Service endpoint**: Enable Microsoft.Storage service endpoint on the VM's subnet, then add a VNet network rule to the storage account allowing that subnet.
2. **Private endpoint**: Create a private endpoint for the storage account in the VM's VNet, giving it a private IP address.

Additionally, you could add the VM's public IP to the storage account's IP network rules (if the VM has a public IP), but VNet-based options are more secure.

### A5
Configure **Customer-Managed Keys (CMK)** — store the encryption key in **Azure Key Vault** and configure the storage account to use it. The storage account needs a **managed identity** to access the Key Vault. For the second layer, enable **Infrastructure encryption** (double encryption). ⚠️ Infrastructure encryption must be enabled **at storage account creation** — it cannot be added later.

### A6
Configure **Azure File Sync**:
1. Create a **Storage Sync Service** in Azure
2. Create an **Azure file share** as the central cloud store
3. Install the **Azure File Sync agent** on each branch office Windows server
4. Register each server with the Storage Sync Service
5. Create a **Sync Group** with the Azure file share as the cloud endpoint
6. Add each server's folder as a **server endpoint**
7. Enable **Cloud Tiering** to keep only frequently used files local (set volume free space policy, e.g., 20% free)

### A7
Three features required:
1. **Blob soft delete** — must be enabled
2. **Change feed** — must be enabled
3. **Blob versioning** — must be enabled

All three must be active before point-in-time restore can be configured.

### A8
**Immediately**:
1. **Regenerate both access keys** (key1 and key2) to invalidate the compromised key
2. Update all applications using those keys with the new keys

**Going forward**:
1. Use **Microsoft Entra ID authorization** instead of access keys
2. Disable shared key access: `Set-AzStorageAccount -AllowSharedKeyAccess $false`
3. Use **Azure Key Vault** to store any secrets/keys
4. Enable **Azure Defender for Storage** for threat detection

### A9
Modify or delete the **stored access policy**. Options:
- Set the policy's **expiry time to a past date**
- **Remove all permissions** from the policy
- **Delete the policy** entirely

Any of these will immediately invalidate all 50 SAS tokens linked to that policy, without needing to regenerate the storage account key.

### A10
Prerequisites:
- **Blob versioning** must be enabled on **both** source and destination accounts
- **Change feed** must be enabled on the **source** account
- Both accounts must be **General-purpose v2** or **Premium block blob** accounts
- Only **block blobs** are supported (not page or append blobs)
- Blob snapshots are **NOT replicated**

### A11
The Archive tier is **offline** — blobs cannot be read or modified directly. The user must **rehydrate** the blob to the Hot or Cool tier:
- **Standard priority**: Up to **15 hours**
- **High priority**: Under **1 hour** (for blobs < 10 GB)

Rehydration creates a copy or changes the tier. The blob remains in Archive until rehydration completes.

### A12
Recommend **Azure Files** because:
- Azure Files supports **SMB protocol** — the app can continue using the same UNC path (`\\server\share`)
- Minimal or no code changes required for the lift-and-shift
- Azure Files can be **mounted as a network drive** on Windows (`net use Z: \\storageaccount.file.core.windows.net\share`)
- Blob Storage requires REST API access and would require significant application changes

### A13
Storage account naming rules:
- **3–24 characters** long
- **Lowercase letters and numbers only** — no hyphens, underscores, uppercase, or special characters
- Must be **globally unique** across all of Azure

"My-Storage-Account" fails: contains hyphens and uppercase. "mystorageaccount123!" fails: contains special character (!).

### A14
Set the storage account's **public network access to Disabled** (or set default network action to Deny with no exceptions). Simply creating a private endpoint does NOT automatically disable public access — you must explicitly disable it. Also configure a **Private DNS Zone** to resolve the storage FQDN to the private IP.

### A15
**10 days ago**: Yes, the container can be **recovered** because it's within the 14-day soft delete retention period. Restore it from the Azure portal (show deleted containers) or via API.

**20 days ago**: No, the container is **permanently deleted** because it exceeds the 14-day retention window and cannot be recovered.
