# 3.5 Practice Questions — Deploy and Manage Compute

> Scenario-based questions to test your understanding. Answers are at the bottom.

---

## Questions

### Q1 — ARM Template Deployment Modes
You have a resource group with 3 VMs and 2 storage accounts. You deploy an ARM template that defines only the 3 VMs using **Complete mode**. What happens to the storage accounts?

---

### Q2 — Availability Sets vs Zones
Your company needs 99.99% SLA for the web tier. Should you use Availability Sets or Availability Zones?

### Q3 — VM Sizing
A developer's B2s VM runs slow during peak hours but fine during off-peak. What is likely happening?

---

### Q4 — VM States and Billing
A developer shuts down a VM from the OS (Start Menu → Shut Down). Will billing stop?

---

### Q5 — Custom VM Image
You need a custom Windows image deployed across 3 regions and multiple subscriptions. What's the recommended approach?

### Q6 — Container Choice
- A: Simple batch script running once daily
- B: Microservices API needing autoscaling
- C: Complex ML pipeline with custom Kubernetes operators

Which Azure container service for each?

### Q7 — App Service Deployment Slots
You swap staging to production, but users see staging data. The database connection string was NOT marked as a slot setting. What went wrong?

---

### Q8 — App Service Scaling
Your web app on **Basic (B1)** plan can't find the autoscale option. Why?

---

### Q9 — Data Disk
You attach a 128 GB disk to a running Windows VM. The disk doesn't appear in File Explorer. What's missing?

---

### Q10 — Bicep vs ARM JSON
Will switching to Bicep affect existing ARM JSON deployments?

---

## Answers

### A1
The 2 storage accounts will be **deleted**. Complete mode deletes resources not in the template. Use **Incremental mode** (default) to avoid this. Always run **What-If** before Complete mode.

### A2
**Availability Zones** — 99.99% SLA vs 99.95% for Availability Sets. Zones distribute VMs across physically separate datacenters.

### A3
**B-series VMs are burstable.** They accumulate CPU credits during low usage and spend them during peaks. When credits are exhausted, the VM is throttled to baseline. Solution: upgrade to D-series (non-burstable) or larger B-series.

### A4
**No.** OS shutdown = **Stopped** state (still allocated, still billed). Must **deallocate** via Azure Portal/CLI/PowerShell to stop compute charges. `az vm deallocate` or `Stop-AzVM` properly deallocates.

### A5
Use **Azure Compute Gallery**: generalize VM (Sysprep) → capture image → create gallery → create image definition → create image version → replicate to 3 regions → share via RBAC.

### A6
- **A**: **ACI** (restart policy = Never) — simple, serverless
- **B**: **Container Apps** — built-in autoscale, traffic splitting
- **C**: **AKS** — full Kubernetes, custom operators

### A7
Connection strings **swap with the app** by default. The staging DB connection followed the code to production. **Fix**: mark database connection string as **"Deployment slot setting"** (sticky to slot).

### A8
Autoscale requires **Standard tier+**. Basic only supports manual scaling. Scale up to S1+ first, then configure autoscale.

### A9
The disk needs to be **initialized, partitioned, and formatted** inside the OS. In Windows: open Disk Management → Initialize → Create volume → Format NTFS → Assign drive letter.

### A10
**No.** Bicep transpiles to ARM JSON — fully compatible. You can use both simultaneously. Decompile JSON to Bicep with `az bicep decompile --file template.json`.
