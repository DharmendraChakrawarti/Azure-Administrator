# 3.4 Create and Configure Azure App Service — AZ-104 Study Notes

---

## Overview

Azure App Service is a fully managed **PaaS** (Platform as a Service) for hosting web applications, REST APIs, and mobile backends. It supports multiple programming languages and frameworks without managing infrastructure.

---

## 3.4.1 Azure App Service Basics

### What is App Service?
- A **fully managed** web hosting platform
- Supports: **.NET, Java, Node.js, Python, PHP, Ruby, Go, custom containers**
- Built-in **CI/CD**, autoscaling, custom domains, SSL, and authentication

### Key Features

| Feature | Description |
|---------|-------------|
| **Multiple languages** | .NET, Java, Node.js, Python, PHP, Ruby |
| **Auto-scaling** | Scale up (bigger plan) or scale out (more instances) |
| **Deployment slots** | Staging environments for zero-downtime deployments |
| **Custom domains** | Map custom DNS names + SSL certificates |
| **Authentication** | Built-in auth with Microsoft Entra ID, Google, Facebook, etc. |
| **Hybrid connections** | Connect to on-premises resources |
| **WebJobs** | Run background tasks |

---

## 3.4.2 App Service Plans

### What is an App Service Plan?
- Defines the **compute resources** for your web app
- Every App Service app runs in an App Service Plan
- Multiple apps can share the **same plan**

### App Service Plan Tiers

| Tier | Description | Scaling | Slots | Features |
|------|-------------|---------|-------|----------|
| **Free (F1)** | Shared compute | None | 0 | Dev/test, 1 GB disk |
| **Shared (D1)** | Shared compute | None | 0 | Custom domains |
| **Basic (B1/B2/B3)** | Dedicated compute | Manual | 0 | Dev/test, dedicated |
| **Standard (S1/S2/S3)** | Dedicated compute | Auto | 5 | Production, autoscale |
| **Premium (P1v3/P2v3/P3v3)** | Dedicated compute | Auto | 20 | Enhanced performance |
| **Isolated (I1v2/I2v2/I3v2)** | Dedicated environment (ASE) | Auto | 20 | Network isolation |

### Plan Selection Guide

| Need | Recommended Tier |
|------|-----------------|
| Development/testing | Free / Basic |
| Production with autoscale | Standard / Premium |
| Deployment slots | Standard+ (5 slots) or Premium+ (20 slots) |
| VNet integration | Standard+ |
| Full network isolation | Isolated (ASE) |
| Custom domains | Shared+ |
| SSL bindings | Basic+ |

### Creating an App Service Plan
```powershell
# Create App Service Plan
New-AzAppServicePlan -ResourceGroupName "myRG" `
  -Name "myPlan" -Location "eastus" `
  -Tier "Standard" -NumberofWorkers 2 -WorkerSize "Small"
```

```bash
# Create App Service Plan
az appservice plan create --name myPlan \
  --resource-group myRG --location eastus \
  --sku S1 --number-of-workers 2
```

### Scaling

#### Scale Up (Vertical) — Change plan tier
```bash
az appservice plan update --name myPlan \
  --resource-group myRG --sku P1v3
```

#### Scale Out (Horizontal) — Add more instances
```bash
# Manual scale
az appservice plan update --name myPlan \
  --resource-group myRG --number-of-workers 5

# Autoscale (Standard tier+)
az monitor autoscale create --resource-group myRG \
  --resource myPlan --resource-type Microsoft.Web/serverfarms \
  --min-count 2 --max-count 10 --count 3
```

> [!NOTE]
> **Scale up** changes the VM size/tier (vertical). **Scale out** adds more instances (horizontal). Autoscale (scale out) requires **Standard tier or higher**.

## 3.4.3 Create and Configure Web Apps

### Creating a Web App — Portal
1. App Services → + Create → Web App
2. **Basics**: Subscription, RG, Name (globally unique), Publish (Code/Container), Runtime stack, Region, App Service Plan
3. **Deployment**: CI/CD configuration
4. **Networking**: VNet integration, private endpoints
5. **Monitoring**: Application Insights
6. **Tags**: Key-value pairs

### PowerShell Example
```powershell
# Create web app
New-AzWebApp -ResourceGroupName "myRG" `
  -Name "mywebapp-unique" `
  -Location "eastus" `
  -AppServicePlan "myPlan"
```

### Azure CLI Example
```bash
# Create web app
az webapp create --resource-group myRG \
  --plan myPlan --name mywebapp-unique \
  --runtime "DOTNET:8.0"

# List runtimes
az webapp list-runtimes --output table
```

### App Settings (Environment Variables)
- **Key-value pairs** available as environment variables to your application
- Override values in config files (e.g., `appsettings.json`)
- **Connection strings** — specifically for database connections

```bash
# Set app setting
az webapp config appsettings set --resource-group myRG \
  --name mywebapp-unique \
  --settings DB_HOST=myserver.database.windows.net DB_NAME=mydb

# Set connection string
az webapp config connection-string set --resource-group myRG \
  --name mywebapp-unique \
  --settings MyDbConnection='Server=...;Database=...' \
  --connection-string-type SQLAzure
```

### General Configuration

| Setting | Description |
|---------|-------------|
| **Stack** | .NET, Java, Node, Python, PHP |
| **Platform** | 32-bit or 64-bit |
| **Always On** | Keep app loaded (Standard+ only) |
| **ARR Affinity** | Session affinity (sticky sessions) |
| **HTTPS Only** | Redirect HTTP to HTTPS |
| **Minimum TLS** | TLS 1.0, 1.1, 1.2 |
| **FTP state** | FTPS Only, Disabled, All |

## 3.4.4 Deployment Methods

### Deployment Options

| Method | Description |
|--------|-------------|
| **Azure DevOps** | CI/CD pipelines |
| **GitHub Actions** | GitHub-integrated CI/CD |
| **Local Git** | Push from local Git repo |
| **FTP/FTPS** | Upload files directly |
| **ZIP Deploy** | Upload ZIP package |
| **CLI/PowerShell** | `az webapp deploy` |
| **VS Code** | Publish from IDE |

### Local Git Deployment
```bash
# Configure local git
az webapp deployment source config-local-git --resource-group myRG \
  --name mywebapp-unique

# Get deployment URL (use as git remote)
az webapp deployment list-publishing-credentials --resource-group myRG \
  --name mywebapp-unique --query scmUri
```

### GitHub Actions Deployment
```bash
# Configure GitHub Actions
az webapp deployment github-actions add --resource-group myRG \
  --name mywebapp-unique --repo "user/repo" --branch main
```

## 3.4.5 Deployment Slots

### What Are Deployment Slots?
- Separate **live environments** for your web app with their own hostnames
- Used for **staging, testing, and zero-downtime deployments**
- **Swap** slots to promote code from staging to production

### Key Properties

| Property | Detail |
|----------|--------|
| **Minimum tier** | Standard (5 slots) or Premium (20 slots) |
| **Hostname** | `appname-slotname.azurewebsites.net` |
| **Configuration** | Each slot has its own settings |
| **Swap** | Zero-downtime swap between slots |
| **Auto swap** | Automatically swap after deployment |
| **Swap with preview** | Multi-phase swap with validation |

### Settings That Do/Don't Swap

| Swapped (follow the app) | NOT Swapped (stay with slot) |
|--------------------------|----------------------------|
| App settings (unless marked "slot setting") | Publishing endpoints |
| Connection strings (unless marked "slot setting") | Custom domain names |
| Handler mappings | TLS/SSL certificates |
| Public certificates | Scale settings |
| WebJobs content | Diagnostic settings |

### Slot Operations
```bash
# Create deployment slot
az webapp deployment slot create --resource-group myRG \
  --name mywebapp-unique --slot staging

# Deploy to slot
az webapp deployment source config-zip --resource-group myRG \
  --name mywebapp-unique --slot staging --src app.zip

# Swap staging → production
az webapp deployment slot swap --resource-group myRG \
  --name mywebapp-unique --slot staging --target-slot production

# Auto swap
az webapp deployment slot auto-swap --resource-group myRG \
  --name mywebapp-unique --slot staging --auto-swap-slot production
```

```powershell
# Swap slots
Switch-AzWebAppSlot -ResourceGroupName "myRG" `
  -Name "mywebapp-unique" `
  -SourceSlotName "staging" `
  -DestinationSlotName "production"
```

> [!TIP]
> Mark settings as **"slot settings"** (sticky to the slot) for environment-specific values like database connection strings for staging vs production databases.

## 3.4.6 Custom Domains and SSL

### Adding a Custom Domain
1. **Verify ownership** — Add a CNAME or TXT record to DNS
2. **Map the domain** — Add custom domain in Azure portal
3. **Bind SSL certificate** — Upload and bind TLS/SSL certificate

### DNS Records

| Record Type | Purpose | Example |
|-------------|---------|---------|
| **CNAME** | Map subdomain to App Service | `www → mywebapp.azurewebsites.net` |
| **A Record** | Map root domain to App Service IP | `@ → IP address` |
| **TXT** | Domain verification | `asuid.www → verification ID` |

### SSL/TLS Options

| Option | Description |
|--------|-------------|
| **App Service Managed Certificate** | Free, auto-renewed, for custom domains |
| **Bring Your Own Certificate** | Upload PFX certificate |
| **App Service Certificate** | Purchase from Azure |
| **Key Vault Certificate** | Store certificate in Key Vault |

```bash
# Add custom domain
az webapp config hostname add --resource-group myRG \
  --webapp-name mywebapp-unique --hostname www.contoso.com

# Upload SSL certificate
az webapp config ssl upload --resource-group myRG \
  --name mywebapp-unique --certificate-file cert.pfx \
  --certificate-password "P@ss"

# Bind SSL
az webapp config ssl bind --resource-group myRG \
  --name mywebapp-unique --certificate-thumbprint <thumbprint> \
  --ssl-type SNI
```

## 3.4.7 Application Insights and Monitoring

### What is Application Insights?
- **APM** (Application Performance Management) service
- Part of **Azure Monitor**
- Tracks requests, failures, dependencies, exceptions, page views

### Key Metrics
- **Response time**, **failure rate**, **availability**, **server requests/sec**
- **Dependency tracking** — calls to databases, APIs, storage
- **Custom events** and **custom metrics**

### Enabling Application Insights
```bash
# Enable Application Insights
az monitor app-insights component create --app myAppInsights \
  --resource-group myRG --location eastus

# Link to web app
az webapp config appsettings set --resource-group myRG \
  --name mywebapp-unique \
  --settings APPINSIGHTS_INSTRUMENTATIONKEY=<instrumentation-key>
```

## 3.4.8 App Service Networking

### Inbound Features

| Feature | Description |
|---------|-------------|
| **App-assigned address** | Dedicated IP for SSL |
| **Access restrictions** | IP-based allow/deny rules |
| **Service endpoints** | Access from specific VNet subnets |
| **Private endpoints** | Access via private IP in VNet |

### Outbound Features

| Feature | Description |
|---------|-------------|
| **VNet Integration** | App connects to resources in a VNet |
| **Hybrid Connections** | Connect to on-premises resources (no VPN needed) |
| **Gateway-required VNet Integration** | Uses VPN gateway for VNet connectivity |

### Access Restrictions
```bash
# Add IP restriction (allow specific IP)
az webapp config access-restriction add --resource-group myRG \
  --name mywebapp-unique --rule-name "AllowOffice" \
  --priority 100 --action Allow --ip-address 203.0.113.0/24

# Add VNet rule
az webapp config access-restriction add --resource-group myRG \
  --name mywebapp-unique --rule-name "AllowVNet" \
  --priority 200 --action Allow \
  --vnet-name myVNet --subnet mySubnet
```

## Exam Tips — Section 3.4

- **App Service Plan** defines compute; multiple apps can share one plan
- Free/Shared tiers = shared infrastructure; Basic+ = dedicated VMs
- **Deployment slots** require **Standard tier+** (5 slots) or **Premium** (20 slots)
- Slot swap is **zero-downtime**; know which settings swap vs. stay
- **Slot settings** (sticky) don't swap — use for environment-specific configs
- **Scale up** = change tier; **Scale out** = add instances; Autoscale = Standard+
- **Always On** keeps app loaded (prevents cold starts) — requires Standard+
- Custom domains need CNAME/A records + domain verification via TXT
- **App Service Managed Certificate** is free but only for custom domains
- **VNet Integration** allows outbound access to VNet resources
- **Access restrictions** control inbound access (IP rules, VNet rules)
- **Private endpoints** provide inbound access via private IP
- Know deployment methods: Git, GitHub Actions, ZIP deploy, FTP
- App settings override application config files as environment variables
- **Application Insights** provides APM — track performance, failures, dependencies
