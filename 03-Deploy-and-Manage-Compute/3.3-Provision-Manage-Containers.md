# 3.3 Provision and Manage Containers — AZ-104 Study Notes

---

## Overview

Azure provides several container services for running containerized applications. For the AZ-104 exam, the primary focus is on **Azure Container Instances (ACI)** and **Azure Container Apps**, with some knowledge of **Azure Kubernetes Service (AKS)**.

---

## 3.3.1 Azure Container Instances (ACI)

### What is ACI?
- The **fastest and simplest** way to run a container in Azure
- **Serverless** — no VM management required
- Pay per **second** of container execution
- Best for: isolated containers, simple apps, task automation, CI/CD build agents

### Key Properties

| Property | Detail |
|----------|--------|
| **OS** | Windows and Linux |
| **Isolation** | Hypervisor-level (not shared kernel) |
| **Startup** | Seconds (no VM provisioning) |
| **Persistent storage** | Azure Files shares |
| **Networking** | Public IP, VNet integration |
| **CPU** | Up to 4 cores per container |
| **Memory** | Up to 16 GB per container |
| **Restart policy** | Always, OnFailure, Never |

### Creating a Container Instance — Portal
1. Container instances → + Create
2. **Basics**: Subscription, RG, Container name, Region, Image source (Quickstart/ACR/Docker Hub/Other), Image, Size
3. **Networking**: Public/Private/None, DNS name label, Ports
4. **Advanced**: Restart policy, Environment variables, Command override

### PowerShell Example
```powershell
# Create container instance
New-AzContainerGroup -ResourceGroupName "myRG" `
  -Name "mycontainer" `
  -Image "mcr.microsoft.com/azuredocs/aci-helloworld" `
  -OsType Linux `
  -DnsNameLabel "mycontainer-demo" `
  -Port @(80) `
  -Cpu 1 -MemoryInGB 1.5 `
  -RestartPolicy "OnFailure"
```

### Azure CLI Example
```bash
# Create container instance
az container create --resource-group myRG \
  --name mycontainer \
  --image mcr.microsoft.com/azuredocs/aci-helloworld \
  --dns-name-label mycontainer-demo \
  --ports 80 \
  --cpu 1 --memory 1.5 \
  --restart-policy OnFailure

# View container logs
az container logs --resource-group myRG --name mycontainer

# Get container details
az container show --resource-group myRG --name mycontainer \
  --query "{FQDN:ipAddress.fqdn, Status:instanceView.state}" --output table

# Execute command in running container
az container exec --resource-group myRG --name mycontainer \
  --exec-command "/bin/bash"

# Delete container
az container delete --resource-group myRG --name mycontainer --yes
```

### Container Groups
- Multiple containers deployed together on the **same host**
- Share lifecycle, resources, local network, and storage volumes
- Similar to a **pod** in Kubernetes
- Defined using **YAML** or **ARM template**

```yaml
# container-group.yaml
apiVersion: '2021-09-01'
location: eastus
name: myContainerGroup
properties:
  containers:
  - name: frontend
    properties:
      image: nginx
      ports:
      - port: 80
      resources:
        requests:
          cpu: 1.0
          memoryInGB: 1.5
  - name: sidecar
    properties:
      image: mcr.microsoft.com/azuredocs/aci-tutorial-sidecar
      resources:
        requests:
          cpu: 0.5
          memoryInGB: 0.5
  osType: Linux
  ipAddress:
    type: Public
    ports:
    - port: 80
type: Microsoft.ContainerInstance/containerGroups
```

```bash
# Deploy container group from YAML
az container create --resource-group myRG --file container-group.yaml
```

### Restart Policies

| Policy | Description |
|--------|-------------|
| **Always** | Containers always restart (default) — long-running services |
| **OnFailure** | Restart only on failure — batch/task processing |
| **Never** | Run once and stop — one-time tasks |

### Mounting Azure Files to ACI
```bash
az container create --resource-group myRG --name mycontainer \
  --image myimage:latest --cpu 1 --memory 1.5 \
  --azure-file-volume-account-name mystorageaccount \
  --azure-file-volume-account-key "<key>" \
  --azure-file-volume-share-name myfileshare \
  --azure-file-volume-mount-path /mnt/data
```

> [!NOTE]
> Azure Files is the only supported persistent storage for ACI. emptyDir volumes are also available but are ephemeral.

## 3.3.2 Azure Container Registry (ACR)

### What is ACR?
- A **private Docker registry** in Azure for storing and managing container images
- Based on Docker Registry 2.0
- Supports Docker images, Helm charts, and OCI artifacts

### ACR Tiers

| Tier | Storage | Throughput | Geo-Replication | Use Case |
|------|---------|-----------|-----------------|----------|
| **Basic** | 10 GB | Low | ❌ | Dev/test |
| **Standard** | 100 GB | Medium | ❌ | Production |
| **Premium** | 500 GB | High | ✅ | Enterprise, multi-region |

### Creating ACR
```bash
# Create ACR
az acr create --resource-group myRG --name myACR --sku Standard

# Login to ACR
az acr login --name myACR

# Build image in ACR (ACR Tasks)
az acr build --registry myACR --image myapp:v1 .

# Push image to ACR
docker tag myapp:latest myACR.azurecr.io/myapp:v1
docker push myACR.azurecr.io/myapp:v1

# List repositories
az acr repository list --name myACR --output table
```

### ACR Authentication

| Method | Description |
|--------|-------------|
| **Azure AD (Individual)** | `az acr login` — developer access |
| **Azure AD (Service Principal)** | App registration with ACR roles |
| **Admin Account** | Shared username/password (not recommended for production) |
| **Managed Identity** | VM/ACI/AKS accesses ACR via managed identity |

### ACR Roles

| Role | Pull Images | Push Images | Delete Images |
|------|------------|------------|---------------|
| **AcrPull** | ✅ | ❌ | ❌ |
| **AcrPush** | ✅ | ✅ | ❌ |
| **AcrDelete** | ❌ | ❌ | ✅ |
| **Contributor** | ✅ | ✅ | ✅ |

## 3.3.3 Azure Container Apps

### What Are Container Apps?
- **Serverless container platform** built on Kubernetes
- No need to manage Kubernetes infrastructure
- Built-in **autoscaling** (including scale to zero), **HTTPS ingress**, **Dapr** integration

### Key Features

| Feature | Detail |
|---------|--------|
| **Scaling** | Scale to zero; KEDA-based autoscaling |
| **Revisions** | Traffic splitting between revisions |
| **Ingress** | Built-in HTTPS with custom domains |
| **Secrets** | Secure management of app secrets |
| **Dapr** | Built-in microservice framework |
| **Networking** | VNet integration, internal/external ingress |

### Container Apps vs ACI vs AKS

| Feature | ACI | Container Apps | AKS |
|---------|-----|---------------|-----|
| **Complexity** | Simplest | Medium | Most complex |
| **Scaling** | Manual | Auto (to zero) | Auto (not to zero by default) |
| **Orchestration** | None | Built-in (managed K8s) | Full Kubernetes |
| **Networking** | Basic | Built-in ingress | Full K8s networking |
| **Use case** | Simple tasks | Microservices, APIs | Complex, full control |

### Creating Container Apps
```bash
# Create Container Apps environment
az containerapp env create --name myEnvironment \
  --resource-group myRG --location eastus

# Create Container App
az containerapp create --name myapp \
  --resource-group myRG \
  --environment myEnvironment \
  --image mcr.microsoft.com/azuredocs/containerapps-helloworld:latest \
  --target-port 80 \
  --ingress external \
  --min-replicas 0 --max-replicas 5
```

## 3.3.4 Azure Kubernetes Service (AKS) Basics

### What is AKS?
- Managed **Kubernetes** service in Azure
- Azure manages the **control plane** (API server, etcd, scheduler)
- You manage the **worker nodes** (agent pools)

### Key AKS Concepts

| Concept | Description |
|---------|-------------|
| **Cluster** | AKS deployment with control plane + node pools |
| **Node Pool** | Group of VMs running containerized workloads |
| **System Node Pool** | Runs critical system pods (CoreDNS, kube-proxy) |
| **User Node Pool** | Runs your application workloads |
| **kubectl** | CLI for interacting with Kubernetes API |

### Creating an AKS Cluster
```bash
# Create AKS cluster
az aks create --resource-group myRG --name myAKSCluster \
  --node-count 2 --node-vm-size Standard_DS2_v2 \
  --generate-ssh-keys --enable-managed-identity

# Get credentials for kubectl
az aks get-credentials --resource-group myRG --name myAKSCluster

# Verify
kubectl get nodes

# Scale node pool
az aks scale --resource-group myRG --name myAKSCluster --node-count 3

# Enable cluster autoscaler
az aks update --resource-group myRG --name myAKSCluster \
  --enable-cluster-autoscaler --min-count 1 --max-count 5
```

### AKS Networking Options

| Option | Description |
|--------|-------------|
| **Kubenet** | Simple networking; pods get IPs from separate address space |
| **Azure CNI** | Pods get IPs from the VNet subnet; advanced integration |
| **Azure CNI Overlay** | Pods get IPs from an overlay network on Azure CNI |

> [!TIP]
> For AZ-104, know the basics of AKS (creation, scaling, node pools). Deep Kubernetes knowledge is tested in **AZ-305** and **AKS-specific certifications**.

## Exam Tips — Section 3.3

- **ACI** is the simplest container option — serverless, pay-per-second
- **Container groups** (multi-container) in ACI are like Kubernetes pods
- ACI restart policies: **Always** (default), **OnFailure**, **Never**
- ACI persistent storage: **Azure Files** only
- **ACR tiers**: Basic (dev) < Standard (prod) < Premium (geo-replication)
- ACR roles: **AcrPull** (read), **AcrPush** (read+write), **AcrDelete** (delete)
- **Container Apps** = serverless + scale to zero + built-in ingress
- **AKS** = full Kubernetes; Azure manages control plane, you manage nodes
- Know when to use each: ACI (simple tasks) vs Container Apps (microservices) vs AKS (full control)
- ACI supports **VNet integration** for private networking
- ACR admin account is **not recommended** for production
