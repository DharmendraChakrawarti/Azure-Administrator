# 3.2 Create and Configure Virtual Machines — AZ-104 Study Notes

---

## Overview

Azure Virtual Machines (VMs) provide **on-demand, scalable computing resources**. VMs are an IaaS offering that gives you full control over the operating system, software, and configuration. Understanding VM creation, sizing, availability, and management is heavily tested on AZ-104.

---

## 3.2.1 Create a Virtual Machine

### Planning Considerations

| Decision | Options |
|----------|---------|
| **Network** | VNet, subnet, NSG, public IP |
| **Name** | Resource naming convention (e.g., `vm-prod-web-eastus-001`) |
| **Location** | Region closest to users; affects available VM sizes |
| **Size** | CPU, RAM, storage, GPU based on workload |
| **OS** | Windows or Linux; marketplace or custom image |
| **Storage** | OS disk + optional data disks; managed disks recommended |

### Creating VM — Portal
1. Virtual machines → + Create → Azure virtual machine
2. **Basics**: Subscription, RG, Name, Region, Availability, Image, Size, Admin credentials
3. **Disks**: OS disk type, data disks
4. **Networking**: VNet, Subnet, Public IP, NSG, Load balancing
5. **Management**: Monitoring, Auto-shutdown, Backup
6. **Advanced**: Extensions, custom data, cloud-init
7. **Tags**: Key-value pairs
8. **Review + Create**

### PowerShell Example
```powershell
# Create a Windows VM
New-AzVm -ResourceGroupName "myRG" `
  -Name "myVM" `
  -Location "eastus" `
  -VirtualNetworkName "myVNet" `
  -SubnetName "default" `
  -SecurityGroupName "myNSG" `
  -PublicIpAddressName "myPublicIP" `
  -Image "Win2022Datacenter" `
  -Size "Standard_DS2_v2" `
  -Credential (Get-Credential)

# Create a Linux VM
New-AzVm -ResourceGroupName "myRG" `
  -Name "myLinuxVM" `
  -Location "eastus" `
  -Image "Ubuntu2204" `
  -Size "Standard_B2s" `
  -GenerateSshKey `
  -SshKeyName "mySSHKey"
```

### Azure CLI Example
```bash
# Create a Windows VM
az vm create --resource-group myRG --name myVM \
  --image Win2022Datacenter --size Standard_DS2_v2 \
  --admin-username azureuser --admin-password 'P@ssw0rd1234!' \
  --vnet-name myVNet --subnet default \
  --public-ip-address myPublicIP --nsg myNSG

# Create a Linux VM
az vm create --resource-group myRG --name myLinuxVM \
  --image Ubuntu2204 --size Standard_B2s \
  --admin-username azureuser --generate-ssh-keys
```

## 3.2.2 VM Sizes and Series

### VM Size Families

| Series | Optimized For | Use Cases |
|--------|--------------|-----------|
| **B-series** | Burstable | Dev/test, small databases, low-traffic web |
| **D-series** | General purpose | Most production workloads |
| **E-series** | Memory optimized | Large databases, in-memory analytics |
| **F-series** | Compute optimized | Batch processing, gaming, analytics |
| **L-series** | Storage optimized | Big data, NoSQL, data warehousing |
| **N-series** | GPU | Machine learning, rendering, video encoding |
| **M-series** | Memory optimized (large) | SAP HANA, very large in-memory workloads |

### Resizing VMs
- Can resize **without redeploying** if new size is available in current hardware cluster
- If size requires a different cluster → VM is **deallocated** and restarted
- Stopped (deallocated) VMs can be resized to any size in the region

```powershell
# List available sizes for a VM
Get-AzVMSize -ResourceGroupName "myRG" -VMName "myVM"

# Resize VM
$vm = Get-AzVM -ResourceGroupName "myRG" -Name "myVM"
$vm.HardwareProfile.VmSize = "Standard_DS3_v2"
Update-AzVM -VM $vm -ResourceGroupName "myRG"
```

```bash
# List available sizes
az vm list-sizes --location eastus --output table

# Resize
az vm resize --resource-group myRG --name myVM --size Standard_DS3_v2
```

> [!TIP]
> To check available sizes in a region: `az vm list-sizes --location eastus`. Not all sizes are available in all regions.

## 3.2.3 VM Availability Options

### Availability Sets
- Protect against **hardware failures** within a datacenter
- Distributes VMs across **Fault Domains** (FDs) and **Update Domains** (UDs)

| Component | Description | Max Count |
|-----------|-------------|-----------|
| **Fault Domain** | Group of VMs sharing common hardware (rack, power, network) | 3 |
| **Update Domain** | Group of VMs that can be rebooted together during maintenance | 20 |

```powershell
# Create availability set
New-AzAvailabilitySet -ResourceGroupName "myRG" `
  -Name "myAvSet" -Location "eastus" `
  -PlatformFaultDomainCount 3 `
  -PlatformUpdateDomainCount 5 `
  -Sku "Aligned"   # Required for managed disks
```

### Availability Zones
- **Physically separate locations** within an Azure region
- Each zone has its own power, cooling, and networking
- Protects against **datacenter-level failures**
- **99.99% SLA** for VMs across availability zones

```powershell
# Create VM in a specific zone
New-AzVm -ResourceGroupName "myRG" -Name "myVM" `
  -Location "eastus" -Zone 1 -Image "Win2022Datacenter" `
  -Size "Standard_DS2_v2" -Credential (Get-Credential)
```

### Availability Comparison

| Feature | Availability Set | Availability Zones |
|---------|-----------------|-------------------|
| **Scope** | Single datacenter | Multiple datacenters |
| **Protects against** | Rack/hardware failure | Datacenter failure |
| **SLA** | 99.95% | 99.99% |
| **Max FDs** | 3 | N/A (zone = FD) |
| **Max UDs** | 20 | N/A |

### Virtual Machine Scale Sets (VMSS)
- Create and manage a group of **identical, load-balanced VMs**
- **Autoscale** based on demand or schedule
- Spread across availability zones

```powershell
# Create VMSS
New-AzVmss -ResourceGroupName "myRG" `
  -VMScaleSetName "myVMSS" `
  -Location "eastus" `
  -VirtualNetworkName "myVNet" `
  -SubnetName "default" `
  -PublicIpAddressName "myPublicIP" `
  -UpgradePolicyMode "Automatic" `
  -InstanceCount 3 `
  -Credential (Get-Credential)
```

```bash
az vmss create --resource-group myRG --name myVMSS \
  --image Ubuntu2204 --instance-count 3 \
  --admin-username azureuser --generate-ssh-keys \
  --upgrade-policy-mode automatic
```

### VMSS Autoscale Rules
- **Metric-based**: CPU, memory, disk, network
- **Schedule-based**: Time-of-day, day-of-week

```bash
# Create autoscale rule
az monitor autoscale create --resource-group myRG \
  --resource myVMSS --resource-type Microsoft.Compute/virtualMachineScaleSets \
  --min-count 2 --max-count 10 --count 3

az monitor autoscale rule create --resource-group myRG \
  --autoscale-name myVMSS --condition "Percentage CPU > 70 avg 5m" \
  --scale out 2
```

> [!NOTE]
> VMSS supports both **Uniform** orchestration (identical VMs, managed by VMSS) and **Flexible** orchestration (mix VM types, higher availability). Flexible is now the recommended mode.

## 3.2.4 Manage VM Disks

### Disk Types

| Disk Type | IOPS (max) | Throughput (max) | Use Case |
|-----------|-----------|-----------------|----------|
| **Standard HDD** | 500 | 60 MB/s | Backup, non-critical |
| **Standard SSD** | 6,000 | 750 MB/s | Web servers, light dev/test |
| **Premium SSD** | 20,000 | 900 MB/s | Production, databases |
| **Premium SSD v2** | 80,000 | 1,200 MB/s | Demanding workloads |
| **Ultra Disk** | 160,000 | 4,000 MB/s | SAP HANA, transaction-heavy DBs |

### Managed vs. Unmanaged Disks

| Feature | Managed Disks | Unmanaged Disks |
|---------|--------------|----------------|
| **Management** | Azure manages storage | You manage storage account |
| **Availability** | 99.999% SLA | Depends on storage account |
| **Recommended** | ✅ Yes | ❌ Legacy |
| **Snapshots** | Built-in | Manual |
| **Encryption** | Automatic SSE | Manual setup |

### Disk Operations
```powershell
# Add data disk to VM
$vm = Get-AzVM -ResourceGroupName "myRG" -Name "myVM"
$diskConfig = New-AzDiskConfig -Location "eastus" `
  -CreateOption Empty -DiskSizeGB 128 -SkuName "Premium_LRS"
$dataDisk = New-AzDisk -ResourceGroupName "myRG" `
  -DiskName "myDataDisk" -Disk $diskConfig
$vm = Add-AzVMDataDisk -VM $vm -Name "myDataDisk" `
  -CreateOption Attach -ManagedDiskId $dataDisk.Id -Lun 0
Update-AzVM -VM $vm -ResourceGroupName "myRG"

# Create snapshot
$snapshotConfig = New-AzSnapshotConfig -Location "eastus" `
  -CreateOption Copy -SourceResourceId $dataDisk.Id
New-AzSnapshot -ResourceGroupName "myRG" `
  -SnapshotName "mySnapshot" -Snapshot $snapshotConfig
```

```bash
# Attach data disk
az vm disk attach --resource-group myRG --vm-name myVM \
  --name myDataDisk --size-gb 128 --sku Premium_LRS --new

# Create snapshot
az snapshot create --resource-group myRG --name mySnapshot \
  --source myDataDisk
```

### Azure Disk Encryption (ADE)
- Encrypts OS and data disks using **BitLocker** (Windows) or **DM-Crypt** (Linux)
- Keys stored in **Azure Key Vault**
- Requires Key Vault with **disk encryption access policy** enabled

```powershell
# Enable encryption
Set-AzVMDiskEncryptionExtension -ResourceGroupName "myRG" `
  -VMName "myVM" `
  -DiskEncryptionKeyVaultUrl $kvURL `
  -DiskEncryptionKeyVaultId $kvId
```

## 3.2.5 VM Extensions and Custom Script

### What are VM Extensions?
- **Post-deployment configuration** and automation for VMs
- Install software, configure settings, run scripts

### Common Extensions

| Extension | Description |
|-----------|-------------|
| **Custom Script Extension** | Run PowerShell/Bash scripts on VM |
| **DSC Extension** | PowerShell Desired State Configuration |
| **Azure Monitor Agent** | Collect monitoring data |
| **Azure Disk Encryption** | Enable BitLocker/DM-Crypt |
| **Dependency Agent** | Map VM dependencies (for Service Map) |

### Custom Script Extension
```powershell
# Run script on Windows VM
Set-AzVMCustomScriptExtension -ResourceGroupName "myRG" `
  -VMName "myVM" -Location "eastus" `
  -FileUri "https://mystorageaccount.blob.core.windows.net/scripts/install.ps1" `
  -Run "install.ps1" -Name "InstallScript"
```

```bash
# Run script on Linux VM
az vm extension set --resource-group myRG --vm-name myLinuxVM \
  --name customScript --publisher Microsoft.Azure.Extensions \
  --settings '{"fileUris": ["https://raw.githubusercontent.com/.../install.sh"], "commandToExecute": "./install.sh"}'
```

## 3.2.6 VM Management Operations

### VM Power States

| State | Billing | Description |
|-------|---------|-------------|
| **Running** | ✅ Yes | VM is on and running |
| **Stopped** | ✅ Yes | OS shut down, but Azure resources allocated |
| **Stopped (Deallocated)** | ❌ No (compute) | Resources released; only storage billed |
| **Starting** | ✅ Yes | VM is booting up |

```powershell
# Start, stop, restart, deallocate
Start-AzVM -ResourceGroupName "myRG" -Name "myVM"
Stop-AzVM -ResourceGroupName "myRG" -Name "myVM"          # Deallocates
Stop-AzVM -ResourceGroupName "myRG" -Name "myVM" -StayProvisioned  # Stopped (still billed)
Restart-AzVM -ResourceGroupName "myRG" -Name "myVM"
```

```bash
az vm start --resource-group myRG --name myVM
az vm stop --resource-group myRG --name myVM          # Stopped (still billed)
az vm deallocate --resource-group myRG --name myVM    # Deallocated (no compute charge)
az vm restart --resource-group myRG --name myVM
```

### Auto-Shutdown
- Schedule automatic shutdown at a specific time
- Portal: VM → Operations → Auto-shutdown
- Useful for dev/test VMs to save costs

### Redeploy VM
- Moves VM to a new host within Azure
- Useful when you cannot RDP/SSH to the VM
- Causes a **restart** with a new IP (unless static)

```bash
az vm redeploy --resource-group myRG --name myVM
```

> [!WARNING]
> **Stopped** (via OS shutdown) ≠ **Deallocated**. A stopped VM still incurs **compute charges**. Always **deallocate** to stop billing.

## 3.2.7 Generalize and Capture VM Images

### Process to Create a Custom Image
1. **Generalize** the VM (remove machine-specific info)
   - Windows: Run `sysprep` (generalize + shutdown)
   - Linux: Run `sudo waagent -deprovision+user`
2. **Deallocate** and **generalize** via Azure
3. **Capture** the VM as an image

```powershell
# Mark VM as generalized
Stop-AzVM -ResourceGroupName "myRG" -Name "myVM" -Force
Set-AzVm -ResourceGroupName "myRG" -Name "myVM" -Generalized

# Capture image
$vm = Get-AzVM -ResourceGroupName "myRG" -Name "myVM"
$image = New-AzImageConfig -Location "eastus" -SourceVirtualMachineId $vm.Id
New-AzImage -Image $image -ImageName "myCustomImage" -ResourceGroupName "myRG"
```

### Azure Compute Gallery (Shared Image Gallery)
- **Store, version, and share** custom images across subscriptions and regions
- Image definitions → Image versions → replicate across regions

```bash
# Create gallery
az sig create --resource-group myRG --gallery-name myGallery

# Create image definition
az sig image-definition create --resource-group myRG \
  --gallery-name myGallery --gallery-image-definition myImageDef \
  --publisher myPublisher --offer myOffer --sku mySku \
  --os-type Windows --os-state Generalized

# Create image version
az sig image-version create --resource-group myRG \
  --gallery-name myGallery --gallery-image-definition myImageDef \
  --gallery-image-version 1.0.0 \
  --managed-image "/subscriptions/{sub-id}/resourceGroups/myRG/providers/Microsoft.Compute/images/myCustomImage"
```

> [!CAUTION]
> Once a VM is **generalized**, it **cannot be started again**. The original VM is no longer usable. Always create the image from a generalized VM.

## Exam Tips — Section 3.2

- Know VM creation parameters: Name, Region, Size, Image, VNet, NSG, Admin credentials
- **Availability Sets**: FDs (max 3) + UDs (max 20) = protect against rack/maintenance failures
- **Availability Zones**: Protect against datacenter failure; 99.99% SLA
- **VMSS** for autoscaling identical VMs; supports metric-based and schedule-based scaling
- **Stopped** (OS shutdown) ≠ **Deallocated** — only deallocated stops compute billing
- Know disk types: Standard HDD < Standard SSD < Premium SSD < Ultra Disk
- **Managed Disks** are recommended; unmanaged disks are legacy
- **Azure Disk Encryption**: BitLocker (Windows) + DM-Crypt (Linux), keys in Key Vault
- **Custom Script Extension** runs post-deployment scripts
- **Generalized VMs** cannot be restarted — plan accordingly
- **Azure Compute Gallery** for sharing images across subscriptions/regions
- **Resizing** may require deallocation if new size isn't in current cluster
- **B-series** VMs are burstable — great for variable workloads (dev/test)
