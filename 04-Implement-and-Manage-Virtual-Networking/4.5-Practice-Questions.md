# 4.5 Practice Questions — Virtual Networking

> Scenario-based questions to test your understanding. Answers are at the bottom.

---

## Questions

### Q1 — NSG Rule Evaluation
An NSG has these inbound rules:
- Priority 100: Allow TCP 80 from Any
- Priority 200: Deny TCP 80 from 10.0.1.0/24
- Priority 300: Allow TCP 443 from Any

A request comes from IP 10.0.1.5 on port 80. Is it allowed or denied?

---

### Q2 — VNet Peering
You have VNet-A peered with VNet-B, and VNet-B peered with VNet-C. Can VMs in VNet-A communicate with VMs in VNet-C?

---

### Q3 — Subnet IP Addresses
You create a subnet with address range 10.0.1.0/24 (256 IPs). How many usable IPs are available for resources?

---

### Q4 — Azure Bastion
Your security team says all RDP access to VMs must go through Azure Bastion, and no VM should have a public IP. What are the Bastion subnet requirements?

---

### Q5 — Load Balancer vs Application Gateway
You need to load balance HTTPS traffic with URL-based routing (/api/* → API pool, /web/* → Web pool). Should you use Azure Load Balancer or Application Gateway?

---

### Q6 — VPN Gateway
You need to connect your on-premises network (192.168.0.0/16) to Azure VNet (10.0.0.0/16) with encrypted traffic. What resources do you need to create?

---

### Q7 — Traffic Manager Routing
Your app is deployed in East US and West Europe. You want users to connect to the region closest to them for lowest latency. Which Traffic Manager routing method should you use?

---

### Q8 — Network Watcher
A VM can't connect to a database on port 1433 in another subnet. You suspect an NSG is blocking traffic. Which Network Watcher tool should you use first?

---

### Q9 — NSG Flow Logs
Your security team wants to audit all network traffic through your NSGs, including bytes transferred and flow states. Which flow log version do you need?

---

### Q10 — Private DNS Zone
You create a Private DNS Zone and link it to VNet-A with auto-registration enabled. A VM is created in VNet-A. What happens automatically?

---

## Answers

### A1
**Allowed.** NSG rules are evaluated by priority (lowest number first). Rule 100 (Allow TCP 80 from Any) matches first and allows the traffic. Rule 200 (Deny from 10.0.1.0/24) is never reached because rule 100 already matched.

### A2
**No.** VNet peering is **NOT transitive**. A↔B and B↔C does NOT mean A↔C. You must create a separate peering between VNet-A and VNet-C, or use a hub-spoke topology with gateway transit.

### A3
**251 usable IPs.** Azure reserves 5 IPs per subnet: .0 (network), .1 (gateway), .2 and .3 (DNS), .255 (broadcast). So 256 - 5 = **251**.

### A4
Bastion requirements:
- Subnet must be named **`AzureBastionSubnet`**
- Minimum subnet size: **/26** (64 IPs)
- Bastion gets a **Standard SKU public IP** (VMs do NOT need public IPs)
- Bastion connects over **HTTPS (port 443)** — no RDP/SSH ports exposed

### A5
**Application Gateway** — it's a Layer 7 (HTTP/HTTPS) load balancer that supports **URL path-based routing**, SSL termination, and WAF. Azure Load Balancer is Layer 4 (TCP/UDP) and cannot inspect HTTP URLs.

### A6
Four resources needed:
1. **GatewaySubnet** in your VNet (recommended /27+)
2. **VPN Gateway** (virtual network gateway) in the GatewaySubnet
3. **Local Network Gateway** — represents the on-prem VPN device (public IP + on-prem address space 192.168.0.0/16)
4. **Connection** — links VPN Gateway to Local Network Gateway with a shared key

### A7
**Performance** routing method — routes users to the endpoint with the lowest network latency. Traffic Manager measures latency from Azure regions to Internet egress points and directs users accordingly.

### A8
Use **IP Flow Verify** — it tests whether a specific packet (source IP, dest IP, protocol, port) is allowed or denied by NSG rules, and tells you **which NSG rule** is responsible. If NSG isn't the issue, use **Next Hop** to check routing.

### A9
**Version 2.** Flow log version 1 provides basic 5-tuple info (source/dest IP, ports, protocol, action). Version 2 adds **bytes transferred, packets, and flow state** (Begin, Continuing, End).

### A10
An **A record** is automatically created in the Private DNS Zone for the new VM (e.g., `myvm.private.contoso.com → 10.0.1.5`). This is the auto-registration feature. Note: only **one** private DNS zone per VNet for auto-registration, but multiple for resolution.
