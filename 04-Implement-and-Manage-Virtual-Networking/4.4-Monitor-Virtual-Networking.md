# 4.4 Monitor Virtual Networking — AZ-104 Study Notes

---

## Overview

Monitoring virtual networking in Azure involves using tools to **diagnose connectivity issues**, **analyze traffic flows**, and **monitor network performance**. Key tools include Azure Network Watcher, NSG flow logs, and Connection Monitor.

---

## 4.4.1 Azure Network Watcher

### What is Network Watcher?
- A **regional service** that provides network diagnostic and visualization tools
- Automatically enabled in each region where you have Azure resources
- Provides tools for monitoring, diagnosing, and gaining insights into network health

### Network Watcher Tools

| Category | Tool | Description |
|----------|------|-------------|
| **Monitoring** | Connection Monitor | End-to-end connectivity monitoring |
| **Monitoring** | Topology | Visual network topology map |
| **Diagnostics** | IP Flow Verify | Test if a packet is allowed/denied by NSG |
| **Diagnostics** | Next Hop | Determine the next hop for a packet |
| **Diagnostics** | Effective Security Rules | View effective NSG rules on a NIC |
| **Diagnostics** | VPN Troubleshoot | Diagnose VPN gateway/connection issues |
| **Diagnostics** | Packet Capture | Capture network packets on a VM |
| **Diagnostics** | Connection Troubleshoot | One-time connectivity check |
| **Logging** | NSG Flow Logs | Log network traffic through NSGs |
| **Traffic** | Traffic Analytics | Analyze NSG flow logs with Log Analytics |

## 4.4.2 IP Flow Verify

### What Does It Do?
- Tests whether a **specific packet** is allowed or denied by NSG rules
- Identifies **which NSG rule** is allowing or blocking traffic
- Useful for troubleshooting connectivity issues

### Parameters Required
- **VM** and **NIC**
- **Direction**: Inbound or Outbound
- **Protocol**: TCP or UDP
- **Local IP and Port**
- **Remote IP and Port**

### Usage
```bash
# IP Flow Verify
az network watcher test-ip-flow --resource-group myRG \
  --vm myVM --direction Inbound --protocol TCP \
  --local 10.0.1.4:80 --remote 203.0.113.5:* \
  --nic myNIC

# Returns:
# Access: Allow/Deny
# Rule Name: <which NSG rule matched>
```

```powershell
Test-AzNetworkWatcherIPFlow -NetworkWatcher $nw `
  -TargetVirtualMachineId $vm.Id `
  -Direction Inbound -Protocol TCP `
  -LocalIPAddress "10.0.1.4" -LocalPort 80 `
  -RemoteIPAddress "203.0.113.5" -RemotePort "*"
```

> [!TIP]
> Use **IP Flow Verify** when you can't connect to a VM — it quickly tells you if an NSG rule is blocking the traffic and which specific rule is responsible.

## 4.4.3 Next Hop

### What Does It Do?
- Shows the **next hop type and IP address** for traffic from a VM
- Helps identify **routing issues** (wrong UDR, missing route)
- Shows whether traffic goes through VNet, Internet, NVA, or is dropped

### Next Hop Types
- Virtual Network
- Internet
- Virtual Network Gateway
- Virtual Appliance (with IP)
- None (traffic is dropped)

```bash
# Check next hop
az network watcher show-next-hop --resource-group myRG \
  --vm myVM --source-ip 10.0.1.4 --dest-ip 10.0.2.5

# Returns: nextHopType and nextHopIpAddress
```

```powershell
Get-AzNetworkWatcherNextHop -NetworkWatcher $nw `
  -TargetVirtualMachineId $vm.Id `
  -SourceIPAddress "10.0.1.4" `
  -DestinationIPAddress "10.0.2.5"
```

## 4.4.4 Effective Security Rules

### What Does It Do?
- Shows the **aggregated, effective NSG rules** applied to a NIC
- Includes rules from both **subnet NSG** and **NIC NSG**
- Helps identify why traffic is being allowed or blocked

```bash
# View effective security rules
az network nic list-effective-nsg --resource-group myRG --name myNIC
```

```powershell
Get-AzEffectiveNetworkSecurityGroup -NetworkInterfaceName "myNIC" `
  -ResourceGroupName "myRG"
```

## 4.4.5 Connection Troubleshoot

### What Does It Do?
- Performs a **one-time connectivity check** between two endpoints
- Tests TCP connectivity from a VM to a destination (VM, URI, IP:port)
- Shows: reachability status, latency, probes sent/failed, hops traversed

```bash
# Test connectivity
az network watcher test-connectivity --resource-group myRG \
  --source-resource myVM --dest-address 10.0.2.5 \
  --dest-port 443 --protocol TCP
```

```powershell
Test-AzNetworkWatcherConnectivity -NetworkWatcher $nw `
  -SourceId $sourceVM.Id `
  -DestinationAddress "10.0.2.5" `
  -DestinationPort 443
```

## 4.4.6 Connection Monitor

### What Does It Do?
- **Continuous monitoring** of connectivity between endpoints
- Tracks **reachability, latency, and network topology changes** over time
- Supports Azure VMs, on-premises, and external endpoints
- Replaces the legacy Network Performance Monitor

### Creating Connection Monitor
```bash
az network watcher connection-monitor create \
  --resource-group myRG --name myConnectionMonitor \
  --location eastus \
  --test-group-name "WebToDb" \
  --endpoint-source-name "WebVM" --endpoint-source-resource-id <vm-id> \
  --endpoint-dest-name "DbServer" --endpoint-dest-address 10.0.2.5 \
  --test-config-name "TCPTest" --protocol TCP --tcp-port 1433 \
  --test-frequency 30
```

### Key Features

| Feature | Detail |
|---------|--------|
| **Sources** | Azure VMs, on-prem machines (with Log Analytics agent) |
| **Destinations** | Azure VMs, external endpoints, URLs |
| **Protocols** | TCP, HTTP, HTTPS, ICMP |
| **Alerts** | Integration with Azure Monitor alerts |
| **Dashboards** | Built-in topology view and metrics |

## 4.4.7 NSG Flow Logs

### What Are NSG Flow Logs?
- Record information about **IP traffic flowing through an NSG**
- Stored in an **Azure Storage account** as JSON
- Essential for security auditing, compliance, and traffic analysis

### Flow Log Data (Per Flow)

| Field | Description |
|-------|-------------|
| **Time** | Timestamp |
| **Source/Destination IP** | Endpoints |
| **Source/Destination Port** | Ports |
| **Protocol** | TCP or UDP |
| **Direction** | Inbound or Outbound |
| **Action** | Allow or Deny |
| **Bytes/Packets** | Volume (version 2) |
| **Flow State** | Begin, Continuing, End (version 2) |

### Flow Log Versions

| Version | Features |
|---------|----------|
| **Version 1** | Basic flow info (5-tuple + action) |
| **Version 2** | + Bytes, packets, flow state |

### Enabling NSG Flow Logs
```bash
# Enable flow logs
az network watcher flow-log create --resource-group myRG \
  --name myFlowLog --nsg myNSG \
  --storage-account mystorageaccount \
  --enabled true --format JSON --log-version 2 \
  --retention 30 --location eastus

# Enable Traffic Analytics
az network watcher flow-log create --resource-group myRG \
  --name myFlowLog --nsg myNSG \
  --storage-account mystorageaccount \
  --enabled true --format JSON --log-version 2 \
  --traffic-analytics true \
  --workspace myLogAnalyticsWorkspace \
  --interval 10
```

### Storage Location
- Flow logs are stored in the storage account at:
  ```
  insights-logs-networksecuritygroupflowevent/
    resourceId=.../NETWORKSECURITYGROUPS/<nsg-name>/
      y=<year>/m=<month>/d=<day>/h=<hour>/m=00/
        macAddress=<mac>/PT1H.json
  ```

> [!NOTE]
> NSG flow logs are stored in a **storage account**. To analyze them effectively, enable **Traffic Analytics** which processes the logs and provides visual dashboards in Log Analytics.

## 4.4.8 Traffic Analytics

### What is Traffic Analytics?
- A **cloud-based solution** that provides visibility into user and application activity in cloud networks
- Processes **NSG flow logs** and provides insights through **Log Analytics**
- Visualizes traffic patterns, security threats, and performance issues

### Key Insights

| Insight | Description |
|---------|-------------|
| **Traffic distribution** | Flows across regions, VNets, subnets |
| **Most communicating hosts** | Top talkers (VMs with most traffic) |
| **Blocked traffic** | NSG rules blocking traffic |
| **Security threats** | Malicious IPs, open ports |
| **Geographic distribution** | Traffic source countries |
| **Port utilization** | Most used ports |

### Requirements
1. **NSG flow logs** enabled (version 2 recommended)
2. **Storage account** for flow log storage
3. **Log Analytics workspace** for Traffic Analytics processing
4. **Network Watcher** enabled in the region

### Processing Intervals
- **Every 10 minutes** or **every 60 minutes** (configurable)

## 4.4.9 Packet Capture

### What is Packet Capture?
- Captures **network packets** on a VM for deep analysis
- Saves captures to **VM storage** or **Azure Storage account**
- Requires the **Network Watcher Agent** VM extension

### Creating Packet Capture
```bash
# Install Network Watcher Agent extension (if not installed)
az vm extension set --resource-group myRG --vm-name myVM \
  --name NetworkWatcherAgentWindows \
  --publisher Microsoft.Azure.NetworkWatcher

# Create packet capture
az network watcher packet-capture create --resource-group myRG \
  --vm myVM --name myCapture \
  --storage-account mystorageaccount \
  --time-limit 60 --packet-limit 100 \
  --filters '[{"protocol":"TCP","localPort":"80"}]'

# Stop capture
az network watcher packet-capture stop --name myCapture \
  --location eastus

# Delete capture
az network watcher packet-capture delete --name myCapture \
  --location eastus
```

### Key Settings

| Setting | Description |
|---------|-------------|
| **Time limit** | Max capture duration (seconds, default 18000 = 5 hours) |
| **Byte limit** | Max bytes per packet to capture |
| **Packet limit** | Max number of packets |
| **Filters** | Filter by protocol, IP, port |
| **Storage** | Local VM file or Azure Storage account |

## 4.4.10 Network Topology

### What Does It Do?
- Provides a **visual diagram** of your network resources and their relationships
- Shows VNets, subnets, NICs, VMs, NSGs, public IPs, load balancers
- Helps understand resource dependencies

```bash
# Get topology
az network watcher show-topology --resource-group myRG --location eastus
```

## Exam Tips — Section 4.4

- **Network Watcher** is automatically enabled per region; regional service
- **IP Flow Verify**: Quick test — is traffic allowed/denied by NSG? Which rule?
- **Next Hop**: What's the next hop for traffic? Identifies routing issues
- **Connection Troubleshoot**: One-time TCP connectivity test between endpoints
- **Connection Monitor**: Continuous monitoring of connectivity and latency
- **Effective Security Rules**: Aggregated NSG rules on a NIC (subnet + NIC NSGs combined)
- **NSG Flow Logs**: Record IP traffic through NSGs; stored in Storage account as JSON
  - Version 1: basic 5-tuple; Version 2: + bytes, packets, flow state
- **Traffic Analytics**: Analyzes NSG flow logs using Log Analytics; requires workspace
- **Packet Capture**: Deep packet inspection; requires Network Watcher Agent extension
- Know when to use each tool:
  - Can't connect → **IP Flow Verify** (NSG issue) or **Next Hop** (routing issue)
  - Ongoing monitoring → **Connection Monitor**
  - Security audit → **NSG Flow Logs** + **Traffic Analytics**
  - Deep analysis → **Packet Capture**
