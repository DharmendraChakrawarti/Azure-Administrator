# 4.3 Configure Load Balancing — AZ-104 Study Notes

---

## Overview

Azure provides several load balancing services to distribute traffic across resources for **high availability**, **scalability**, and **performance**. This section covers Azure Load Balancer, Application Gateway, Front Door, and Traffic Manager.

---

## 4.3.1 Azure Load Balancer

### What is Azure Load Balancer?
- A **Layer 4 (TCP/UDP)** load balancer
- Distributes inbound traffic across backend pool instances
- Supports both **public** (internet-facing) and **internal** (private) scenarios

### Load Balancer SKUs

| Feature | Basic | Standard |
|---------|-------|----------|
| **Backend pool size** | Up to 300 | Up to 1,000 |
| **Health probes** | TCP, HTTP | TCP, HTTP, HTTPS |
| **Availability Zones** | ❌ | ✅ Zone-redundant |
| **SLA** | None | 99.99% |
| **Secure by default** | ❌ Open | ✅ Closed (need NSG) |
| **Outbound rules** | ❌ | ✅ |
| **HA Ports** | ❌ | ✅ |
| **Cost** | Free | Per rule + per GB |

> [!IMPORTANT]
> **Standard SKU** is recommended for production. Basic SKU is being retired. Standard is **secure by default** — you must add NSG rules to allow traffic.

### Load Balancer Components

| Component | Description |
|-----------|-------------|
| **Frontend IP** | The IP address that clients connect to |
| **Backend Pool** | Group of VMs or VMSS instances |
| **Health Probe** | Checks if backend instances are healthy |
| **Load Balancing Rule** | Maps frontend IP:port to backend pool:port |
| **Inbound NAT Rule** | Forwards traffic to a specific VM (port forwarding) |
| **Outbound Rule** | Controls SNAT for outbound connections (Standard only) |

### Distribution Algorithms

| Algorithm | Description |
|-----------|-------------|
| **Five-tuple hash** (default) | Source IP, Source Port, Dest IP, Dest Port, Protocol |
| **Source IP affinity** (Session persistence) | Source IP + Dest IP (2-tuple) or + Protocol (3-tuple) |

### Session Persistence Options

| Mode | Hash | Description |
|------|------|-------------|
| **None** | 5-tuple | Default; no affinity |
| **Client IP** | 2-tuple | Same client → same backend |
| **Client IP and Protocol** | 3-tuple | Same client + protocol → same backend |

### Creating a Load Balancer
```bash
# Create public load balancer
az network lb create --resource-group myRG \
  --name myLoadBalancer --sku Standard \
  --frontend-ip-name myFrontend \
  --public-ip-address myPublicIP --location eastus

# Create backend pool
az network lb address-pool create --resource-group myRG \
  --lb-name myLoadBalancer --name myBackendPool

# Create health probe
az network lb probe create --resource-group myRG \
  --lb-name myLoadBalancer --name myHealthProbe \
  --protocol TCP --port 80 --interval 15 --threshold 2

# Create load balancing rule
az network lb rule create --resource-group myRG \
  --lb-name myLoadBalancer --name myLBRule \
  --protocol TCP --frontend-port 80 --backend-port 80 \
  --frontend-ip-name myFrontend \
  --backend-pool-name myBackendPool \
  --probe-name myHealthProbe \
  --idle-timeout 4 --enable-tcp-reset true

# Create inbound NAT rule (port forwarding)
az network lb inbound-nat-rule create --resource-group myRG \
  --lb-name myLoadBalancer --name SSHtoVM1 \
  --protocol TCP --frontend-port 50001 --backend-port 22 \
  --frontend-ip-name myFrontend
```

### Internal Load Balancer
```bash
# Create internal load balancer
az network lb create --resource-group myRG \
  --name myInternalLB --sku Standard \
  --vnet-name myVNet --subnet AppSubnet \
  --frontend-ip-name myPrivateFrontend \
  --private-ip-address 10.0.2.100
```

### Health Probes

| Protocol | How It Works |
|----------|-------------|
| **TCP** | Successful TCP handshake = healthy |
| **HTTP/HTTPS** | HTTP 200 response = healthy |
| **Custom path** | HTTP probe with specific URL path |

```bash
# HTTP health probe with custom path
az network lb probe create --resource-group myRG \
  --lb-name myLoadBalancer --name httpProbe \
  --protocol HTTP --port 80 --path "/health" \
  --interval 10 --threshold 3
```

> [!TIP]
> Set the health probe **interval** low enough to detect failures quickly, but not so low that it generates excessive traffic. Default: 15 seconds, threshold: 2 failures.

## 4.3.2 Azure Application Gateway

### What is Application Gateway?
- A **Layer 7 (HTTP/HTTPS)** load balancer
- Provides **URL-based routing**, **SSL termination**, **WAF**, **session affinity**, and more

### Application Gateway vs Load Balancer

| Feature | Load Balancer | Application Gateway |
|---------|--------------|-------------------|
| **Layer** | Layer 4 (TCP/UDP) | Layer 7 (HTTP/HTTPS) |
| **SSL termination** | ❌ | ✅ |
| **URL routing** | ❌ | ✅ |
| **WAF** | ❌ | ✅ |
| **Cookie affinity** | ❌ | ✅ |
| **Rewrites** | ❌ | ✅ Header/URL rewrite |
| **WebSocket** | ❌ | ✅ |

### Application Gateway Components

| Component | Description |
|-----------|-------------|
| **Frontend IP** | Public or private IP |
| **Listener** | Receives incoming requests (port + protocol + hostname) |
| **Routing Rule** | Maps listener to backend pool via HTTP settings |
| **Backend Pool** | VMs, VMSS, App Services, IPs, FQDNs |
| **HTTP Settings** | Port, protocol, cookie affinity, connection draining |
| **Health Probe** | Custom health probes (HTTP/HTTPS) |

### Application Gateway SKUs

| SKU | Features |
|-----|----------|
| **Standard V2** | Autoscaling, zone redundancy, static VIP, header rewrite |
| **WAF V2** | Everything in Standard V2 + Web Application Firewall |

### Routing Types

| Type | Description |
|------|-------------|
| **Basic** | All requests to one backend pool |
| **Path-based** | Route based on URL path (`/images/*` → image pool, `/api/*` → API pool) |
| **Multi-site** | Route based on hostname (`app1.contoso.com` vs `app2.contoso.com`) |

### Creating Application Gateway
```bash
# Create Application Gateway (simplified)
az network application-gateway create --resource-group myRG \
  --name myAppGateway --sku Standard_v2 \
  --vnet-name myVNet --subnet AGSubnet \
  --public-ip-address myAGPublicIP \
  --http-settings-port 80 --http-settings-protocol Http \
  --frontend-port 80 --capacity 2

# Add path-based routing
az network application-gateway url-path-map create --resource-group myRG \
  --gateway-name myAppGateway --name myPathMap \
  --paths "/api/*" --address-pool apiPool \
  --http-settings apiSettings --default-address-pool defaultPool \
  --default-http-settings defaultSettings
```

### Web Application Firewall (WAF)
- Protects web applications from common exploits (OWASP Top 10)
- **Detection mode**: Log only
- **Prevention mode**: Block malicious requests
- Custom rules + managed rule sets (OWASP 3.2)

## 4.3.3 Azure Front Door

### What is Azure Front Door?
- A **global, scalable** entry point for web applications
- Combines CDN, load balancing, WAF, and SSL at the **edge**
- Uses **Microsoft's global network** for acceleration

### Key Features

| Feature | Description |
|---------|-------------|
| **Global load balancing** | Route traffic to closest/healthiest backend |
| **URL-based routing** | Path-based and host-based routing |
| **SSL offloading** | Terminate SSL at the edge |
| **Caching** | CDN capabilities at the edge |
| **WAF** | Global WAF protection |
| **Session affinity** | Sticky sessions |
| **URL rewrite/redirect** | Modify URLs at the edge |

### Front Door vs Application Gateway

| Feature | Front Door | Application Gateway |
|---------|-----------|-------------------|
| **Scope** | Global (multi-region) | Regional (single region) |
| **Layer** | Layer 7 | Layer 7 |
| **Use case** | Multi-region apps | Single-region apps |
| **CDN** | ✅ Built-in | ❌ |

## 4.3.4 Azure Traffic Manager

### What is Traffic Manager?
- **DNS-based** traffic load balancer
- Distributes traffic across global Azure endpoints
- Works at the **DNS level** — clients connect directly to endpoints

### Routing Methods

| Method | Description |
|--------|-------------|
| **Priority** | Primary/failover — route to highest priority; failover on failure |
| **Weighted** | Distribute traffic based on weights (e.g., 70/30 split) |
| **Performance** | Route to nearest endpoint (lowest latency) |
| **Geographic** | Route based on user's geographic location |
| **Multivalue** | Return multiple healthy endpoints in DNS response |
| **Subnet** | Route based on client's source IP/subnet |

### Key Properties

| Property | Detail |
|----------|--------|
| **Layer** | DNS-based (Layer 7 — DNS) |
| **Protocol** | Works with any protocol (HTTP, TCP, etc.) |
| **Health checks** | HTTP/HTTPS/TCP probes |
| **TTL** | DNS TTL controls caching (default 60 seconds) |
| **Nested profiles** | Combine routing methods |

### Creating Traffic Manager
```bash
# Create Traffic Manager profile
az network traffic-manager profile create --resource-group myRG \
  --name myTMProfile --routing-method Performance \
  --unique-dns-name myapp --ttl 60 --protocol HTTPS \
  --port 443 --path "/health"

# Add endpoint
az network traffic-manager endpoint create --resource-group myRG \
  --profile-name myTMProfile --type azureEndpoints \
  --name EastUSEndpoint --target-resource-id <webapp-id> \
  --endpoint-status enabled
```

### Endpoint Types

| Type | Description |
|------|-------------|
| **Azure endpoint** | Azure services (Web Apps, Public IPs, etc.) |
| **External endpoint** | IPs or FQDNs outside Azure |
| **Nested endpoint** | Another Traffic Manager profile |

> [!NOTE]
> Traffic Manager operates at the **DNS level** — it does NOT proxy traffic. Clients connect directly to the selected endpoint. This means it cannot do SSL termination or URL path routing.

## 4.3.5 Choosing the Right Load Balancer

### Decision Matrix

| Requirement | Solution |
|-------------|----------|
| **Layer 4 (TCP/UDP)** load balancing | Azure Load Balancer |
| **Layer 7 (HTTP)** within a region | Application Gateway |
| **Layer 7 (HTTP)** across regions + CDN | Azure Front Door |
| **DNS-based** global routing | Traffic Manager |
| **WAF** within a region | Application Gateway WAF |
| **WAF** globally | Azure Front Door WAF |

### Common Combination Patterns

```
Global: Traffic Manager/Front Door
  ├── Region 1: Application Gateway → Backend Pool
  └── Region 2: Application Gateway → Backend Pool

OR

Internal: Internal Load Balancer → App VMs
External: Application Gateway → Internal LB → App VMs
```

## Exam Tips — Section 4.3

- **Load Balancer** = Layer 4 (TCP/UDP); **Application Gateway** = Layer 7 (HTTP/HTTPS)
- **Standard LB** is secure by default (need NSG); Basic LB is open (being deprecated)
- LB distribution: **5-tuple hash** (default) or **session persistence** (2/3-tuple)
- **Health probes**: TCP (LB), HTTP/HTTPS (LB Standard, App Gateway)
- **Application Gateway** supports: URL routing, SSL termination, WAF, cookie affinity
- **WAF** protects against OWASP Top 10 exploits
- **Traffic Manager** = DNS-based, does NOT proxy traffic
- Traffic Manager routing: Priority (failover), Weighted (split), Performance (nearest), Geographic (location)
- **Front Door** = global Layer 7 + CDN + WAF at the edge
- Know when to use each: LB (L4) vs App Gateway (L7 regional) vs Front Door (L7 global) vs Traffic Manager (DNS)
- **Inbound NAT rules** on LB = port forwarding to specific VMs
- **Internal LB** uses a private IP from the subnet
- **HA Ports** (Standard LB only) load balance all TCP/UDP ports simultaneously
