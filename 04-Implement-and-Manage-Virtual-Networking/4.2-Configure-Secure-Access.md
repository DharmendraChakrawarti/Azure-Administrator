# 4.2 Configure Secure Access to Virtual Networks — AZ-104 Study Notes

---

## Overview

Securing access to Azure virtual networks involves multiple layers: **Network Security Groups (NSGs)**, **Azure Firewall**, **Azure Bastion**, **VPN Gateways**, and **ExpressRoute**. This section covers how to protect and connect to VNet resources securely.

---

## 4.2.1 Network Security Groups (NSGs)

### What is an NSG?
- A **firewall** that filters network traffic to and from Azure resources
- Contains **security rules** that allow or deny inbound and outbound traffic
- Can be associated with **subnets** and/or **NICs**

### NSG Rule Properties

| Property | Description |
|----------|-------------|
| **Priority** | 100–4096; lower number = higher priority |
| **Name** | Unique name for the rule |
| **Port** | Single, range, or `*` (all) |
| **Protocol** | TCP, UDP, ICMP, or `*` (any) |
| **Source/Destination** | IP, CIDR, Service Tag, or ASG |
| **Action** | Allow or Deny |
| **Direction** | Inbound or Outbound |

### Default Rules (Cannot Be Deleted)

#### Inbound Defaults
| Priority | Name | Source | Destination | Action |
|----------|------|--------|-------------|--------|
| 65000 | AllowVnetInBound | VirtualNetwork | VirtualNetwork | Allow |
| 65001 | AllowAzureLoadBalancerInBound | AzureLoadBalancer | Any | Allow |
| 65500 | DenyAllInBound | Any | Any | Deny |

#### Outbound Defaults
| Priority | Name | Source | Destination | Action |
|----------|------|--------|-------------|--------|
| 65000 | AllowVnetOutBound | VirtualNetwork | VirtualNetwork | Allow |
| 65001 | AllowInternetOutBound | Any | Internet | Allow |
| 65500 | DenyAllOutBound | Any | Any | Deny |

### NSG Association
- **Subnet-level NSG**: Applies to ALL resources in the subnet
- **NIC-level NSG**: Applies to the specific VM
- Both can be applied simultaneously — traffic must pass **both** NSGs

```
Inbound: Internet → Subnet NSG → NIC NSG → VM
Outbound: VM → NIC NSG → Subnet NSG → Internet
```

### Creating NSG and Rules
```powershell
# Create NSG
$nsg = New-AzNetworkSecurityGroup -ResourceGroupName "myRG" `
  -Location "eastus" -Name "myNSG"

# Add rule (allow HTTP inbound)
$nsg | Add-AzNetworkSecurityRuleConfig -Name "AllowHTTP" `
  -Priority 100 -Direction Inbound -Access Allow `
  -Protocol TCP -SourceAddressPrefix "*" `
  -SourcePortRange "*" -DestinationAddressPrefix "*" `
  -DestinationPortRange 80
$nsg | Set-AzNetworkSecurityGroup
```

```bash
# Create NSG
az network nsg create --resource-group myRG --name myNSG --location eastus

# Add rule
az network nsg rule create --resource-group myRG --nsg-name myNSG \
  --name AllowHTTP --priority 100 --direction Inbound \
  --access Allow --protocol TCP --source-address-prefixes '*' \
  --source-port-ranges '*' --destination-address-prefixes '*' \
  --destination-port-ranges 80

# Associate NSG with subnet
az network vnet subnet update --resource-group myRG \
  --vnet-name myVNet --name WebSubnet --network-security-group myNSG
```

### Service Tags (Instead of IP Addresses)

| Service Tag | Description |
|-------------|-------------|
| **Internet** | All public internet traffic |
| **VirtualNetwork** | All VNet + peered VNet + VPN traffic |
| **AzureLoadBalancer** | Azure LB health probes |
| **AzureCloud** | All Azure datacenter IPs |
| **Storage** | Azure Storage service IPs |
| **Sql** | Azure SQL Database IPs |
| **AzureMonitor** | Azure Monitor service IPs |

### Application Security Groups (ASGs)

- **Group VMs logically** (e.g., WebServers, DbServers)
- Use ASGs as source/destination in NSG rules instead of IP addresses
- VMs can be in multiple ASGs

```bash
# Create ASGs
az network asg create --resource-group myRG --name WebServers
az network asg create --resource-group myRG --name DbServers

# Associate VM NIC with ASG
az network nic ip-config update --resource-group myRG \
  --nic-name myNIC --name ipconfig1 \
  --application-security-groups WebServers

# NSG rule using ASGs
az network nsg rule create --resource-group myRG --nsg-name myNSG \
  --name AllowWebToDb --priority 200 --direction Inbound \
  --access Allow --protocol TCP \
  --source-asgs WebServers --destination-asgs DbServers \
  --source-port-ranges '*' --destination-port-ranges 1433
```

> [!TIP]
> Use **ASGs** for readable, maintainable NSG rules. Instead of `allow 10.0.1.5 → 10.0.2.10:1433`, write `allow WebServers → DbServers:1433`.

## 4.2.2 Azure Bastion

### What is Azure Bastion?
- A **PaaS service** for secure RDP/SSH access to VMs **without public IPs**
- Connects via the **Azure portal** over TLS (HTTPS, port 443)
- No need to expose RDP (3389) or SSH (22) to the internet

### Key Features

| Feature | Detail |
|---------|--------|
| **Connection** | Via Azure portal browser (no client needed) |
| **Protocol** | RDP/SSH over HTTPS (port 443) |
| **Public IP** | On Bastion, NOT on VMs |
| **NSG** | No RDP/SSH inbound rules needed on VMs |
| **Subnet** | Must be named `AzureBastionSubnet` (min /26) |
| **Peered VNets** | Can connect to VMs in peered VNets |

### Bastion SKUs

| SKU | Features |
|-----|----------|
| **Basic** | Manual scaling (2 instances), browser-based access |
| **Standard** | Host scaling, native client support, IP-based connection, shareable link |

### Creating Azure Bastion
```bash
# Create AzureBastionSubnet
az network vnet subnet create --resource-group myRG \
  --vnet-name myVNet --name AzureBastionSubnet \
  --address-prefixes 10.0.255.0/26

# Create public IP for Bastion
az network public-ip create --resource-group myRG \
  --name myBastionIP --sku Standard --allocation-method Static

# Create Bastion
az network bastion create --resource-group myRG \
  --name myBastion --public-ip-address myBastionIP \
  --vnet-name myVNet --sku Standard --location eastus
```

> [!IMPORTANT]
> Azure Bastion eliminates the need for **public IPs on VMs** and **NSG rules for RDP/SSH**. It's the recommended way to access VMs securely.

## 4.2.3 VPN Gateway

### What is a VPN Gateway?
- A specific type of **virtual network gateway** that sends encrypted traffic between:
  - Azure VNet ↔ On-premises network (**Site-to-Site**)
  - Azure VNet ↔ Individual client (**Point-to-Site**)
  - Azure VNet ↔ Azure VNet (**VNet-to-VNet**)

### VPN Types

| Type | Description |
|------|-------------|
| **Site-to-Site (S2S)** | IPsec/IKE tunnel between Azure and on-prem VPN device |
| **Point-to-Site (P2S)** | Individual client connects to Azure VNet |
| **VNet-to-VNet** | IPsec tunnel between two Azure VNets |

### Gateway SKUs

| SKU | Tunnels | Throughput | Use Case |
|-----|---------|-----------|----------|
| **Basic** | 10 S2S | 100 Mbps | Dev/test (deprecated) |
| **VpnGw1** | 30 S2S | 650 Mbps | Small production |
| **VpnGw2** | 30 S2S | 1 Gbps | Medium production |
| **VpnGw3** | 30 S2S | 1.25 Gbps | Large production |
| **VpnGw4** | 100 S2S | 5 Gbps | High performance |
| **VpnGw5** | 100 S2S | 10 Gbps | Maximum performance |

### VPN Gateway Requirements
1. **GatewaySubnet** — dedicated subnet in VNet (recommended /27 or larger)
2. **Public IP address** — for the gateway
3. **Local Network Gateway** — represents the on-premises VPN device
4. **Connection** — links the VPN gateway to the local network gateway

### Creating S2S VPN
```bash
# Create GatewaySubnet
az network vnet subnet create --resource-group myRG \
  --vnet-name myVNet --name GatewaySubnet --address-prefixes 10.0.254.0/27

# Create public IP for gateway
az network public-ip create --resource-group myRG \
  --name myGatewayIP --sku Standard --allocation-method Static

# Create VPN gateway (takes 30-45 minutes)
az network vnet-gateway create --resource-group myRG \
  --name myVPNGateway --vnet myVNet --gateway-type Vpn \
  --vpn-type RouteBased --sku VpnGw1 --public-ip-address myGatewayIP

# Create local network gateway (on-prem VPN device)
az network local-gateway create --resource-group myRG \
  --name myLocalGateway --gateway-ip-address 203.0.113.1 \
  --local-address-prefixes 192.168.0.0/16

# Create connection
az network vpn-connection create --resource-group myRG \
  --name S2SConnection --vnet-gateway1 myVPNGateway \
  --local-gateway2 myLocalGateway \
  --shared-key "MySharedKey123"
```

### VPN Types: Policy vs Route Based

| Feature | Policy-Based | Route-Based |
|---------|-------------|------------|
| **IKE version** | IKEv1 | IKEv2 |
| **Tunnels** | 1 | Multiple |
| **Coexist with ExpressRoute** | ❌ | ✅ |
| **P2S support** | ❌ | ✅ |
| **Recommended** | ❌ Legacy | ✅ Yes |

### Point-to-Site VPN
- Individual clients connect to VNet
- Authentication: Certificate, RADIUS, Microsoft Entra ID
- Protocols: OpenVPN, SSTP, IKEv2

```bash
# Configure P2S
az network vnet-gateway update --resource-group myRG \
  --name myVPNGateway \
  --address-prefixes 172.16.0.0/24 \
  --client-protocol OpenVPN
```

## 4.2.4 ExpressRoute

### What is ExpressRoute?
- A **private, dedicated connection** between your on-premises network and Azure
- Does NOT go over the public internet
- Provided by a **connectivity provider** (telecom/ISP)

### ExpressRoute vs VPN Gateway

| Feature | ExpressRoute | VPN Gateway |
|---------|-------------|-------------|
| **Connection** | Private (via provider) | Public internet (encrypted) |
| **Bandwidth** | 50 Mbps – 10 Gbps | Up to 10 Gbps |
| **Latency** | Lower, predictable | Variable |
| **Reliability** | Built-in redundancy | Depends on internet |
| **Cost** | Higher | Lower |
| **Use case** | Enterprise, high bandwidth | Smaller, cost-sensitive |

### ExpressRoute Peering Types

| Peering | Description |
|---------|-------------|
| **Azure Private Peering** | Connect to VNets (VMs, ILBs) |
| **Microsoft Peering** | Connect to Microsoft 365, Dynamics, Azure PaaS |

> [!NOTE]
> **Azure public peering** has been deprecated. Use **Microsoft peering** to access Azure PaaS services via ExpressRoute.

## 4.2.5 Azure Firewall

### What is Azure Firewall?
- A **managed, cloud-based firewall** service
- Stateful firewall-as-a-service with built-in high availability
- Centralized network/application rules with logging

### Key Features

| Feature | Detail |
|---------|--------|
| **Availability** | Built-in high availability + availability zones |
| **FQDN filtering** | Allow/deny traffic to fully qualified domain names |
| **Threat intelligence** | Block known malicious IPs/domains |
| **NAT rules** | DNAT inbound traffic to backend services |
| **Network rules** | Filter by IP, port, protocol |
| **Application rules** | Filter by FQDN (HTTP/S, MSSQL) |
| **Forced tunneling** | Route internet traffic through on-prem firewall |

### Rule Processing Order
1. **NAT rules** (DNAT) — for inbound traffic
2. **Network rules** — by priority
3. **Application rules** — by priority

### Azure Firewall SKUs

| SKU | Features |
|-----|----------|
| **Standard** | L3-L7 filtering, threat intelligence |
| **Premium** | TLS inspection, IDPS, URL filtering, web categories |
| **Basic** | For SMBs; limited features, lower cost |

```bash
# Create Azure Firewall
az network firewall create --resource-group myRG \
  --name myFirewall --location eastus --sku AZFW_VNet --tier Standard

# Create network rule
az network firewall network-rule create --resource-group myRG \
  --firewall-name myFirewall --collection-name AllowWeb \
  --name AllowHTTPS --protocols TCP \
  --source-addresses 10.0.1.0/24 --destination-addresses '*' \
  --destination-ports 443 --action Allow --priority 200
```

## Exam Tips — Section 4.2

- **NSG rules** are evaluated by priority (lowest number = first evaluated)
- Know the **3 default inbound and 3 default outbound** NSG rules
- NSGs can be applied to **subnets AND NICs** — traffic must pass both
- **ASGs** group VMs for readable NSG rules (no IP addresses needed)
- **Service Tags** replace IP ranges for Azure services in NSG rules
- **Azure Bastion** = secure RDP/SSH over HTTPS; no public IP on VMs; needs `AzureBastionSubnet` (/26 min)
- **VPN Gateway** needs `GatewaySubnet`; takes 30–45 min to create
- **Route-Based VPN** is recommended over Policy-Based
- **P2S VPN** supports: certificates, RADIUS, Microsoft Entra ID auth
- **S2S VPN** needs: VPN Gateway + Local Network Gateway + Connection + shared key
- **ExpressRoute** = private, dedicated connection; NOT over internet
- ExpressRoute peering: **Private** (VNets) and **Microsoft** (M365, PaaS)
- **Azure Firewall** rule processing: NAT → Network → Application rules
- Azure Firewall requires `AzureFirewallSubnet` (/26 min)
