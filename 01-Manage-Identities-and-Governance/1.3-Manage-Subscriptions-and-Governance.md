# 1.3 Manage Subscriptions and Governance — AZ-104 Study Notes

---

## Overview

Managing Azure subscriptions and governance is about **controlling, organizing, and optimizing** your Azure environment. This section covers policies, locks, tags, resource groups, subscriptions, costs, and management groups.

---

## 1.3.1 Configure Azure Policies

### What is Azure Policy?
- A service to **create, assign, and manage policies** that enforce rules over your resources
- Ensures resources stay **compliant** with corporate standards and SLAs
- Evaluates resources and highlights those that are **non-compliant**
- Can **prevent** non-compliant resources from being created

### Key Concepts

| Concept | Description |
|---------|-------------|
| **Policy Definition** | A rule (JSON) defining what to evaluate and what action to take |
| **Policy Assignment** | A policy definition applied to a specific scope |
| **Policy Parameters** | Values passed into a policy to make it reusable |
| **Initiative Definition** | A group/collection of policy definitions |
| **Initiative Assignment** | An initiative applied to a specific scope |
| **Compliance** | The state of resources meeting policy conditions |

### Policy Effects (in evaluation order)

| Effect | Description |
|--------|-------------|
| **Disabled** | Policy is turned off; not evaluated |
| **Append** | Adds fields to the resource (e.g., add a tag) |
| **Modify** | Add, update, or remove properties/tags on a resource |
| **Deny** | Prevents the resource from being created/updated |
| **Audit** | Creates a warning event in activity log but doesn't block |
| **AuditIfNotExists** | Audit if a related resource doesn't exist |
| **DeployIfNotExists** | Deploy a related resource if it doesn't exist (requires managed identity) |
| **DenyAction** | Block specific actions on resources (e.g., block delete) |

### Built-in Policy Examples
- **Allowed locations** — Restrict where resources can be deployed
- **Allowed VM SKUs** — Restrict VM sizes
- **Require tag and its value** — Enforce tagging
- **Not allowed resource types** — Block specific resource types
- **Inherit a tag from the resource group** — Auto-tag resources

### Creating a Policy — Portal
1. **Policy** → Definitions → + Policy definition
2. Define: Name, Description, Category
3. Write the **policy rule** (JSON) with conditions and effect
4. Save

### Assigning a Policy — Portal
1. **Policy** → Assignments → Assign policy
2. Select **Scope** (Management group / Subscription / Resource group)
3. Select **Policy definition**
4. Set **Parameters** if applicable
5. Set **Exclusions** (specific scopes to exempt)
6. Set **Non-compliance message**
7. Enable **Remediation** for DeployIfNotExists / Modify effects

### PowerShell Examples
```powershell
# Get a built-in policy definition
$definition = Get-AzPolicyDefinition | Where-Object {
  $_.Properties.DisplayName -eq "Allowed locations"
}

# Assign the policy
$rg = Get-AzResourceGroup -Name "myRG"
$params = @{ listOfAllowedLocations = @("eastus", "westus", "centralus") }

New-AzPolicyAssignment -Name "AllowedLocationsPolicy" `
  -DisplayName "Restrict to US regions" `
  -PolicyDefinition $definition `
  -Scope $rg.ResourceId `
  -PolicyParameterObject $params
```

### Azure CLI Examples
```bash
# Assign a built-in policy
az policy assignment create \
  --name "AllowedLocations" \
  --display-name "Restrict to US regions" \
  --policy "/providers/Microsoft.Authorization/policyDefinitions/{built-in-id}" \
  --scope "/subscriptions/{sub-id}/resourceGroups/myRG" \
  --params '{"listOfAllowedLocations": {"value": ["eastus", "westus"]}}'
```

### Policy Initiatives (Definitions Groups)
- A **collection of policies** grouped together for a single objective
- Example: "Enable Monitoring in Azure Security Center" bundles multiple audit policies
- Assign an initiative instead of individual policies to simplify management

```powershell
# Create initiative
$policies = @(
  @{ policyDefinitionId = "/providers/.../policyDefinitions/policy1" },
  @{ policyDefinitionId = "/providers/.../policyDefinitions/policy2" }
)
New-AzPolicySetDefinition -Name "MyInitiative" `
  -DisplayName "My Security Initiative" `
  -PolicyDefinition ($policies | ConvertTo-Json -Depth 5)
```

### Policy Evaluation
- New policies are evaluated within **~30 minutes** of assignment
- Existing resources are evaluated in a **compliance scan** (runs every 24 hours)
- Trigger on-demand scan:
```bash
az policy state trigger-scan --resource-group "myRG"
```

> [!NOTE]
> Azure Policy evaluates **both new AND existing** resources. Existing non-compliant resources are flagged but not automatically fixed unless using **Modify** or **DeployIfNotExists** with remediation.

### Remediation
- For **DeployIfNotExists** and **Modify** policies, you can create **remediation tasks**
- A **managed identity** is automatically created for the assignment
- Remediation can be run manually or triggered automatically

## 1.3.2 Configure Resource Locks

### What are Resource Locks?
- Prevent accidental **deletion or modification** of critical resources
- Apply to **all users and roles** — even Owners
- Override RBAC permissions

### Lock Types

| Lock Type | Can Read? | Can Update? | Can Delete? |
|-----------|-----------|-------------|-------------|
| **ReadOnly (CanNotModify)** | ✅ Yes | ❌ No | ❌ No |
| **Delete (CanNotDelete)** | ✅ Yes | ✅ Yes | ❌ No |

### Lock Inheritance
- Locks set on a **parent scope** are inherited by **child resources**
- The **most restrictive** lock takes precedence
- Example: Delete lock on RG → all resources in RG inherit Delete lock

### Managing Locks — Portal
Resource/RG/Subscription → Settings → Locks → Add

### PowerShell Examples
```powershell
# Add a delete lock to a resource group
New-AzResourceLock -LockName "CanNotDeleteRG" `
  -LockLevel CanNotDelete `
  -ResourceGroupName "ProductionRG" `
  -LockNotes "Critical production resources"

# Add a read-only lock to a specific resource
New-AzResourceLock -LockName "ReadOnlyStorage" `
  -LockLevel ReadOnly `
  -ResourceGroupName "ProductionRG" `
  -ResourceName "prodstorage" `
  -ResourceType "Microsoft.Storage/storageAccounts"

# Remove a lock
Remove-AzResourceLock -LockName "CanNotDeleteRG" -ResourceGroupName "ProductionRG"
```

### Azure CLI Examples
```bash
# Create a delete lock
az lock create --name "CanNotDeleteRG" \
  --lock-type CanNotDelete \
  --resource-group "ProductionRG"

# List locks
az lock list --resource-group "ProductionRG"

# Delete a lock
az lock delete --name "CanNotDeleteRG" --resource-group "ProductionRG"
```

> [!WARNING]
> A **ReadOnly** lock on a **storage account** prevents listing keys — this can break apps that use storage keys. A ReadOnly lock on a **resource group** prevents creating new resources in it.

### Who Can Manage Locks?
- Only users with `Microsoft.Authorization/locks/*` actions
- Built-in roles: **Owner** and **User Access Administrator**

## 1.3.3 Apply and Manage Tags on Resources

### What are Tags?
- **Key-value pairs** applied to resources, resource groups, and subscriptions
- Used for **logical organization**, cost tracking, and governance
- **Not inherited** by default (use Azure Policy to enforce inheritance)

### Tag Limits

| Limit | Value |
|-------|-------|
| Tags per resource/RG/subscription | **50** |
| Tag name max length | **512** characters |
| Tag value max length | **256** characters |
| Tag name | Case-**insensitive** for operations, case-**sensitive** for display |

### Common Tag Examples

| Tag Name | Example Value | Purpose |
|----------|---------------|---------|
| `Environment` | Production, Dev, Test | Identify environment |
| `CostCenter` | 12345 | Track costs |
| `Owner` | john@contoso.com | Identify responsible person |
| `Department` | Engineering | Organizational unit |
| `Application` | WebApp-Frontend | App identification |
| `CreatedDate` | 2024-01-15 | Track creation |

### Managing Tags — Portal
Resource → Tags → Add key-value pairs → Save

### PowerShell Examples
```powershell
# Add/update tags on a resource group (merge with existing)
$tags = @{ Environment = "Production"; CostCenter = "12345" }
Update-AzTag -ResourceId "/subscriptions/{sub-id}/resourceGroups/myRG" `
  -Tag $tags -Operation Merge

# Replace all tags (destructive — removes existing tags)
Update-AzTag -ResourceId "/subscriptions/{sub-id}/resourceGroups/myRG" `
  -Tag $tags -Operation Replace

# Remove specific tags
Update-AzTag -ResourceId "/subscriptions/{sub-id}/resourceGroups/myRG" `
  -Tag @{ Environment = "" } -Operation Delete

# Get tags
(Get-AzResourceGroup -Name "myRG").Tags
```

### Azure CLI Examples
```bash
# Add/update tags (merge)
az tag update --resource-id "/subscriptions/{sub-id}/resourceGroups/myRG" \
  --operation Merge --tags Environment=Production CostCenter=12345

# Replace all tags
az tag update --resource-id "/subscriptions/{sub-id}/resourceGroups/myRG" \
  --operation Replace --tags NewTag=NewValue
```

### Enforcing Tags with Azure Policy
- **"Require a tag and its value on resources"** — Denies resource creation without the tag
- **"Inherit a tag from the resource group"** — Auto-applies RG tags to child resources (Modify effect)
- **"Require a tag on resource groups"** — Ensures all RGs have specific tags

> [!TIP]
> Tags are **NOT inherited** from RG to resources by default. Use **Azure Policy with Modify effect** to auto-inherit tags from the resource group.

## 1.3.4 Manage Resource Groups

### What is a Resource Group?
- A **container** that holds related Azure resources
- Every Azure resource must belong to **exactly one** resource group
- Resource groups store **metadata** about the resources

### Key Rules

| Rule | Detail |
|------|--------|
| **Region** | RG has a location (for metadata), but resources can be in different regions |
| **Movement** | Resources can be **moved** between RGs (with some limitations) |
| **Deletion** | Deleting a RG **deletes ALL resources** inside it |
| **Nesting** | RGs **cannot be nested** |
| **RBAC** | Permissions can be set at the RG level, inherited by resources |
| **Locks** | Locks on RG are inherited by all resources |

### Creating Resource Groups
```powershell
# PowerShell
New-AzResourceGroup -Name "myRG" -Location "eastus" `
  -Tag @{ Environment = "Production" }
```
```bash
# CLI
az group create --name myRG --location eastus --tags Environment=Production
```

### Moving Resources Between RGs
- Not all resource types support move
- Source and destination RGs are **locked during the move**
- Resources remain **available** during the move (some with brief downtime)

```powershell
# PowerShell
$resource = Get-AzResource -ResourceGroupName "SourceRG" -ResourceName "myVM"
Move-AzResource -DestinationResourceGroupName "DestRG" `
  -ResourceId $resource.ResourceId
```
```bash
# CLI
az resource move --destination-group "DestRG" \
  --ids "/subscriptions/{sub-id}/resourceGroups/SourceRG/providers/Microsoft.Compute/virtualMachines/myVM"
```

### Deleting Resource Groups
```powershell
Remove-AzResourceGroup -Name "myRG" -Force
```
```bash
az group delete --name myRG --yes --no-wait
```

> [!CAUTION]
> Deleting a resource group **permanently deletes ALL resources** inside it. This action is **irreversible**.

## 1.3.5 Manage Subscriptions

### What is an Azure Subscription?
- A **billing and access boundary** for Azure resources
- Linked to an **Azure AD tenant**
- Every resource belongs to exactly one subscription

### Subscription Types

| Type | Description |
|------|-------------|
| **Enterprise Agreement (EA)** | Large orgs with Microsoft contract |
| **Pay-As-You-Go** | Charged monthly based on usage |
| **Free Trial** | $200 credit for 30 days + 12 months free services |
| **CSP (Cloud Solution Provider)** | Through Microsoft partner |
| **Student** | $100 credit for students |
| **MSDN/Visual Studio** | Monthly credits for Visual Studio subscribers |

### Subscription Limits (Quotas)
- Default limits exist (can request increase):
  - VMs per region: 25,000
  - Storage accounts per region: 250
  - Resource groups per subscription: 980
  - VNets per region: 1,000

### Managing Subscription Administrators
```powershell
# Add a subscription-level role
New-AzRoleAssignment -SignInName "user@contoso.com" `
  -RoleDefinitionName "Owner" `
  -Scope "/subscriptions/{sub-id}"
```

### Transferring Subscriptions
- Can transfer **billing ownership** to another account
- Subscription moves to the new account's Azure AD tenant
- RBAC role assignments, policies, and locks are **NOT transferred**

### Creating Additional Subscriptions
- Use multiple subscriptions to:
  - **Separate environments**: Dev, Test, Production
  - **Organizational structure**: Per department/team
  - **Billing**: Separate cost tracking
  - **Subscription limits**: Work around quotas

## 1.3.6 Manage Costs

### Azure Cost Management + Billing

Key tools:

| Tool | Description |
|------|-------------|
| **Cost Analysis** | Visualize and explore spending patterns |
| **Budgets** | Set spending limits with alerts |
| **Advisor Recommendations** | Cost optimization suggestions |
| **Exports** | Schedule cost data exports to storage |
| **Alerts** | Notifications when spending thresholds are reached |

### Cost Analysis — Portal
Cost Management → Cost analysis
- View by: Accumulated / Daily / Resource / Service / Tag
- Filter by: Resource group, Resource type, Location, Tag
- Timeframe: Last 7 days, This month, Custom range

### Creating Budgets
```
Cost Management → Budgets → Add
1. Name the budget
2. Set amount ($) and reset period (Monthly/Quarterly/Annual)
3. Set alert conditions:
   - % of budget threshold (e.g., 50%, 75%, 90%, 100%)
   - Email recipients
   - Action groups (for automated responses)
```

### Cost Optimization Best Practices
1. **Reserved Instances** — Commit 1-3 years for up to 72% savings
2. **Azure Hybrid Benefit** — Use existing Windows Server/SQL licenses
3. **Spot VMs** — Use unused capacity at deep discounts
4. **Right-size resources** — Downscale underutilized VMs
5. **Auto-shutdown** — Schedule dev/test VMs to stop off-hours
6. **Azure Advisor** — Review cost recommendations
7. **Tags** — Tag resources for cost allocation and tracking

### Azure Advisor Cost Recommendations
- Identifies **idle/underutilized** resources
- Suggests **VM resizing**
- Recommends **reserved instances**
- Identifies **unassociated public IPs** and **Express Route circuits**

> [!TIP]
> **Budgets do NOT stop spending** — they only send **alerts**. Use alerts + **action groups** to trigger automation (e.g., shutdown VMs) if needed.

## 1.3.7 Configure Management Groups

### What are Management Groups?
- **Containers** above subscriptions for **organizing** subscriptions hierarchically
- Apply **governance** (policies, RBAC) across multiple subscriptions at once
- Up to **6 levels of depth** (not including root or subscription level)

### Hierarchy Structure
```
Root Management Group (Tenant Root Group)
├── Management Group A
│   ├── Subscription 1
│   └── Subscription 2
├── Management Group B
│   ├── Management Group B1
│   │   └── Subscription 3
│   └── Subscription 4
└── Subscription 5
```

### Key Facts

| Property | Detail |
|----------|--------|
| **Default root group** | Created automatically, named "Tenant Root Group" |
| **Groups per tenant** | 10,000 max |
| **Depth** | 6 levels (not counting root or subscription) |
| **Parents** | Each group/subscription has exactly **one parent** |
| **Children** | Each group can have **multiple children** |
| **Inheritance** | RBAC and Policy assignments inherit downward |

### Creating Management Groups — Portal
Management groups → + Add management group → Enter ID and display name

### PowerShell / CLI
```powershell
# PowerShell
New-AzManagementGroup -GroupName "ProductionMG" `
  -DisplayName "Production" `
  -ParentId "/providers/Microsoft.Management/managementGroups/RootMG"
```
```bash
# CLI
az account management-group create \
  --name "ProductionMG" \
  --display-name "Production" \
  --parent "/providers/Microsoft.Management/managementGroups/RootMG"
```

### Moving Subscriptions to Management Groups
```powershell
New-AzManagementGroupSubscription -GroupId "ProductionMG" `
  -SubscriptionId "{sub-id}"
```
```bash
az account management-group subscription add \
  --name "ProductionMG" \
  --subscription "{sub-id}"
```

### Governance Inheritance
- **Azure Policy** assigned at MG level → applies to all child subscriptions/RGs/resources
- **RBAC** assigned at MG level → inherited by all children
- Example: Assign "Allowed locations" policy at MG → all subscriptions restricted

### Root Management Group
- **Cannot be moved or deleted**
- All other management groups and subscriptions are descendants
- Global Admin must **elevate access** to manage the root group initially

> [!IMPORTANT]
> The **root management group** cannot be moved or deleted. Any policy or RBAC assignment at the root level affects the **entire tenant**.

## Exam Tips — Section 1.3

- **Azure Policy effects order**: Disabled → Append → Modify → Deny → Audit → AuditIfNotExists → DeployIfNotExists
- **Policy ≠ RBAC**: Policy enforces *what* resources look like; RBAC controls *who* can do *what*
- **Resource locks** override RBAC — even Owners are blocked
- **ReadOnly lock** → can read, cannot modify or delete
- **Delete lock** → can read and modify, cannot delete
- **Tags are NOT inherited** — use Policy (Modify effect) to auto-inherit
- **50 tags max** per resource/RG/subscription
- **Resource groups** cannot be nested; resources can be moved between them
- **Deleting a RG** deletes everything inside
- **Budgets** are alerts only — they don't stop spending
- **Management groups** → 6 levels deep, 10,000 per tenant
- **Root management group** cannot be moved or deleted
- **Policy compliance scan** runs every 24 hours automatically
- **DeployIfNotExists** and **Modify** require a **managed identity** for remediation
