# 1.1 Manage Azure AD Objects — AZ-104 Study Notes

---

## Overview

Every user who needs access to Azure resources needs an **Azure user account**. A user account contains all the information needed to **authenticate** the user during sign-on. Once authenticated, Azure AD builds an **access token** to authorize the user and determine:
- What **resources** they can access
- What **actions** they can do with those resources

> [!IMPORTANT]
> Azure Active Directory (Azure AD) has been **renamed to Microsoft Entra ID**. The exam may reference either name.

---

## 1.1.1 Azure AD Fundamentals

### What is Azure AD / Microsoft Entra ID?
- Microsoft's **cloud-based identity and access management** service
- Helps employees sign in and access resources in:
  - **External resources**: Microsoft 365, Azure portal, SaaS applications
  - **Internal resources**: Apps on your corporate network, custom cloud apps

### Key Concepts

| Concept | Description |
|---------|-------------|
| **Tenant** | A dedicated instance of Azure AD representing an organization |
| **Directory** | Each Azure tenant has a dedicated Azure AD directory |
| **Subscription** | Used to pay for Azure services; linked to a tenant |
| **Identity** | An object that can be authenticated (user, app, server) |
| **Account** | An identity that has data associated with it |

### Azure AD vs. On-Premises AD DS

| Feature | Azure AD | AD DS |
|---------|----------|-------|
| **Structure** | Flat (no OUs or GPOs) | Hierarchical (OUs, forests, domains) |
| **Querying** | REST API over HTTP/HTTPS | LDAP |
| **Authentication** | SAML, WS-Federation, OpenID Connect, OAuth 2.0 | Kerberos |
| **Federation** | Built-in federation services | Requires AD FS |

## 1.1.2 Create Users and Groups

### User Types

| Type | Description |
|------|-------------|
| **Member** | Native to the Azure AD directory |
| **Guest** | Invited from outside the organization (B2B) |

### Three Ways to Add Users
1. **Azure Portal** → Azure AD → Users → New user
2. **PowerShell**: `New-AzureADUser`
3. **Azure CLI**: `az ad user create`

### Required Properties for New User
- **Display Name** — e.g., "John Smith"
- **User Principal Name (UPN)** — e.g., `jsmith@contoso.onmicrosoft.com`
- **Password** — Auto-generated or manually set

### PowerShell Example
```powershell
$PasswordProfile = New-Object -TypeName Microsoft.Open.AzureAD.Model.PasswordProfile
$PasswordProfile.Password = "P@ssw0rd1234!"
$PasswordProfile.ForceChangePasswordNextLogin = $true

New-AzureADUser -DisplayName "John Smith" `
  -UserPrincipalName "jsmith@contoso.onmicrosoft.com" `
  -AccountEnabled $true `
  -PasswordProfile $PasswordProfile `
  -MailNickName "jsmith"
```

### Deleting Users
- Deleted users can be **restored within 30 days**
- After 30 days → permanently deleted
- PowerShell: `Remove-AzureADUser -ObjectId <UPN>`

### Group Types

| Group Type | Description |
|------------|-------------|
| **Security** | Manage access to shared resources (most common) |
| **Microsoft 365** | Collaboration — shared mailbox, calendar, files, SharePoint |

### Membership Types

| Membership Type | Description |
|-----------------|-------------|
| **Assigned** | Manually add/remove specific users |
| **Dynamic User** | Auto add/remove based on user attribute rules |
| **Dynamic Device** | Auto add/remove based on device attributes (Security groups only) |

### PowerShell — Create Groups
```powershell
# Security group
New-AzureADGroup -DisplayName "IT Admins" `
  -SecurityEnabled $true -MailEnabled $false -MailNickName "ITAdmins"

# Microsoft 365 dynamic group
New-AzureADMSGroup -DisplayName "Marketing Team" `
  -GroupTypes "DynamicMembership","Unified" `
  -SecurityEnabled $false -MailEnabled $true -MailNickName "Marketing" `
  -MembershipRule '(user.department -eq "Marketing")' `
  -MembershipRuleProcessingState "On"
```

> [!NOTE]
> **Dynamic membership** requires **Azure AD Premium P1 or P2** license.

## 1.1.3 Create Administrative Units

### What Are Administrative Units (AUs)?
- A container that **restricts permissions** to a specific portion of your organization
- Used to **delegate admin roles** with a narrower scope
- Think of them as **OUs in cloud AD**

### Key Facts

| Property | Detail |
|----------|--------|
| **Can contain** | Users, Groups, Devices |
| **License** | Azure AD Premium P1 required for admins |
| **Limit** | Up to 5,000 AUs per tenant |
| **Nesting** | AUs **cannot be nested** |
| **Supported roles** | User Admin, Groups Admin, Helpdesk Admin, Auth Admin, License Admin, Password Admin |

### PowerShell — Create AU
```powershell
New-AzureADMSAdministrativeUnit -DisplayName "Seattle Office" `
  -Description "AU for Seattle office users"
```

### Restricted Management AUs
- Members can **only be managed** by admins scoped to that AU
- Even **Global Admins** are blocked unless explicitly scoped

## 1.1.4 Manage User and Group Properties

### Editable User Properties
- **Identity**: Display name, UPN, User type
- **Job Info**: Title, Department, Manager, Company
- **Contact**: Email, Phone, Address
- **Settings**: Usage location, Sign-in blocked

### PowerShell — Update User/Group
```powershell
Set-AzureADUser -ObjectId "jsmith@contoso.com" `
  -Department "Engineering" -JobTitle "Senior Dev" -UsageLocation "US"

Set-AzureADGroup -ObjectId <GroupId> -DisplayName "New Name"

# Managing membership
Add-AzureADGroupMember -ObjectId <GroupId> -RefObjectId <UserId>
Remove-AzureADGroupMember -ObjectId <GroupId> -MemberId <UserId>
Add-AzureADGroupOwner -ObjectId <GroupId> -RefObjectId <UserId>
```

## 1.1.5 Manage Device Settings

### Device Identity Types

| Type | Description | Ownership |
|------|-------------|-----------|
| **Azure AD Registered** | Personal BYOD devices | Personal |
| **Azure AD Joined** | Signed in with Azure AD account | Organization |
| **Hybrid Azure AD Joined** | Joined to both on-prem AD + Azure AD | Organization |

### Device Settings (Portal: Azure AD → Devices → Device settings)

| Setting | Description |
|---------|-------------|
| Users may join devices to Azure AD | All / Selected / None |
| Require MFA to join devices | Yes / No |
| Max devices per user | Default: 50 |
| Enterprise State Roaming | Sync settings across Windows devices |

## 1.1.6 Perform Bulk User Updates

### Bulk Operations (Portal: Azure AD → Users → Bulk operations)
- **Bulk create** — Upload CSV to create multiple users
- **Bulk invite** — Upload CSV to invite guests
- **Bulk delete** — Upload CSV with user list to delete
- **Download users** — Export to CSV

### PowerShell Example
```powershell
$users = Import-Csv "C:\users.csv"
foreach ($user in $users) {
    $pp = New-Object Microsoft.Open.AzureAD.Model.PasswordProfile
    $pp.Password = $user.Password
    New-AzureADUser -DisplayName $user.DisplayName `
      -UserPrincipalName $user.UPN `
      -AccountEnabled $true -PasswordProfile $pp -MailNickName $user.Nickname
}
```

> [!TIP]
> Download the **template CSV** from the Azure portal to ensure correct formatting.

## 1.1.7 Manage Guest Accounts (B2B)

### What is Azure AD B2B?
- Invite external users as **guest users**
- Guests use their **own credentials** (Google, Microsoft, OTP)

### Guest User Defaults

| Can Do | Cannot Do (default) |
|--------|---------------------|
| Access assigned apps | Enumerate directory users/groups |
| Be member of groups | Read all directory data |
| Be assigned Azure roles | Create apps or register devices |

### Inviting Guests — PowerShell
```powershell
New-AzureADMSInvitation -InvitedUserEmailAddress "partner@external.com" `
  -InvitedUserDisplayName "External Partner" `
  -SendInvitationMessage $true `
  -InviteRedirectUrl "https://myapps.microsoft.com"
```

### External Collaboration Settings (Azure AD → User settings → External collaboration)
- Who can invite: Admins only / Members / Guests / Anyone
- Allow/deny invitations to specific domains

### Access Reviews
- Create **recurring reviews** to ensure guest access is still needed
- Reviewers: Self, group owners, or specific users
- Auto-remove access if not approved

## 1.1.8 Configure Azure AD Join

### Comparison

| Feature | Registered | Joined | Hybrid Joined |
|---------|-----------|--------|---------------|
| **Ownership** | Personal | Org | Org |
| **On-prem AD** | No | No | Yes |
| **SSO to cloud** | Yes | Yes | Yes |
| **SSO to on-prem** | No | Yes | Yes |
| **Management** | MDM | MDM | GPO + MDM |

### How to Join: Settings → Accounts → Access work or school → Connect → Join to Azure AD

### Hybrid Azure AD Join Requirements
1. Azure AD Connect with device sync enabled
2. Service Connection Point (SCP) in on-prem AD
3. Line of sight to domain controller
4. Windows 10/11 or Server 2016+

## 1.1.9 Configure Self-Service Password Reset (SSPR)

### License Requirements

| Feature | License |
|---------|---------|
| Cloud password change | Azure AD Free |
| Cloud SSPR | Premium P1 |
| SSPR + on-prem writeback | Premium P1 + Azure AD Connect |

### Key Settings (Azure AD → Password reset)

| Setting | Options |
|---------|---------|
| SSPR Enabled | None / Selected group / All |
| Methods required | 1 or 2 |
| Available methods | App notification, App code, Email, Mobile phone, Office phone, Security questions |
| Registration | Require on sign-in (Yes/No) |
| Re-confirm period | Default: 180 days |
| Password writeback | Sync resets back to on-prem AD |

### Best Practices
1. Start with a **pilot group** then expand
2. Require **2 authentication methods**
3. Enable **registration enforcement**
4. Enable **notifications** for users and admins
5. Test at https://aka.ms/sspr

> [!WARNING]
> **Admins** always require **two methods** for SSPR. Security questions are **NOT available** for admin accounts.

## Exam Tips — Section 1.1

- Know **Security vs. Microsoft 365** group differences
- **Dynamic groups** → Premium P1
- **Administrative Units** → restrict scope, cannot nest
- **Guest users** → limited permissions by default
- **Hybrid Azure AD Join** → requires Azure AD Connect + SCP
- **SSPR** → Admins need 2 methods; no security questions for admins
- **Deleted users** → restorable within **30 days**
- Bulk operations use **CSV files**
- Key cmdlets: `New-AzureADUser`, `New-AzureADGroup`, `New-AzureADMSGroup`, `Set-AzureADUser`
